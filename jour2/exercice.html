Exercice 1 — Choix d'algorithme

Quatre briefs clients. Pour chacun, identifie : le type de problème ML, l'algorithme recommandé, et la métrique principale d'évaluation. Justifie chaque choix en une ou deux phrases.

Brief A — Mutuelle santé

Une mutuelle veut prédire si un adhérent va résilier son contrat dans les 3 prochains mois, en se basant sur : l'âge, le type de contrat, les remboursements des 12 derniers mois, la fréquence des appels au service client, et la région. Le dataset contient 500 000 adhérents. 8% ont résilié lors de la période précédente.

Brief B — Cabinet médical

Un cabinet de radiologie veut classer automatiquement des comptes-rendus médicaux en texte libre (250 000 documents) selon leur degré d'urgence (urgent / non urgent). Les radiologues ont déjà annoté 2000 documents à la main.

Brief C — Promoteur immobilier

Un promoteur veut estimer le prix de vente de 500 nouvelles maisons individuelles, en se basant sur la surface habitable, la localisation (code postal), le nombre de pièces, la présence d'un jardin, et l'année de construction. Il dispose de 15 000 ventes passées avec les prix réels.

Brief D — Opérateur télécom

Un opérateur télécom veut détecter les appels frauduleux (SIM swapping) en temps réel. Les appels frauduleux représentent moins de 0.01% du trafic. Le modèle doit décider en moins de 50ms par appel. La pénalité pour bloquer un appel légitime (faux positif) est très élevée.

Réponses commentées (à lire après avoir répondu)

Brief A — Mutuelle santé : Churn prediction

  • Type de problème : Classification binaire supervisée (résilié / non résilié)
  • Algorithme recommandé : XGBoost
  • Métrique principale : AUC-ROC (les classes sont déséquilibrées : 8% de résiliations — accuracy serait trompeuse). Compléter avec le rappel si le coût de retenir un client est élevé.
  • Justification : 500 000 lignes, features hétérogènes (montants, catégories, fréquences) → XGBoost. Les relations entre features (ex : adhérent jeune + peu de remboursements + beaucoup d'appels = profil résiliation spécifique) sont non-linéaires. Commencer quand même par une régression logistique pour avoir un baseline et vérifier que les features encodées sont correctes.
  • ⚡ Piège : ne pas utiliser l'accuracy comme métrique principale (8% de résiliations → un modèle qui dit "reste" à chaque fois obtient 92% d'accuracy sans valeur).

Brief B — Cabinet médical : Classification de texte

  • Type de problème : Classification supervisée sur données non structurées (texte)
  • Algorithme recommandé : CamemBERT fine-tuné (HuggingFace Transformers — couvert en Jour 4) — pas XGBoost
  • Métrique principale : Rappel (ne pas manquer un document urgent = faux négatif potentiellement grave). Compléter avec F1 pour l'équilibre.
  • Justification : les données sont du texte libre médical en français — XGBoost ne comprend pas le texte. CamemBERT est pré-entraîné sur le français, il suffit de le fine-tuner sur les 2000 documents annotés. C'est un problème qui sort du ML classique (Jour 5 du programme).
  • ⚡ Piège : ne pas essayer de "transformer le texte en features manuellement" (comptage de mots, TF-IDF) — sous-optimal pour la nuance médicale. Et surtout ne pas utiliser XGBoost sur du texte brut — ça ne fonctionnerait pas.

Brief C — Promoteur immobilier : Estimation de prix

  • Type de problème : Régression supervisée (prédire un nombre continu — le prix)
  • Algorithme recommandé : XGBoost (régresseur), avec régression linéaire comme baseline
  • Métrique principale : RMSE (les grosses erreurs de prix sont coûteuses pour un promoteur — RMSE pénalise davantage les écarts importants). Afficher aussi le MAE pour la communication au métier ("en moyenne, on se trompe de X €").
  • Justification : 15 000 exemples, features hétérogènes (surface numérique, code postal catégoriel, booléen jardin) → pipeline sklearn + XGBoostRegressor. La localisation (code postal) va nécessiter un encodage — OneHotEncoder si peu de codes postaux distincts, encodage par fréquence ou target encoding si beaucoup.
  • ⚡ Piège : ne pas oublier de passer XGBRegressor (pas XGBClassifier) et d'utiliser scoring='neg_root_mean_squared_error' dans crossvalscore.

Brief D — Opérateur télécom : Détection de fraude temps réel

  • Type de problème : Classification binaire supervisée, très déséquilibrée, contrainte de latence
  • Algorithme recommandé : Régression logistique (contrainte 50ms) — pas XGBoost
  • Métrique principale : Précision (la pénalité pour bloquer un appel légitime est très élevée — minimiser les faux positifs). Fixer un seuil de décision bas sur la probabilité.
  • Justification : XGBoost peut prendre 10-50ms par prédiction selon la configuration — risqué pour 50ms max. La régression logistique prédit en moins de 1ms. De plus, avec 0.01% de fraudes, les données sont extrêmement déséquilibrées : utiliser class_weight='balanced' ou sur-échantillonner les fraudes avec SMOTE (Synthetic Minority Over-sampling Technique — génère artificiellement de nouveaux exemples de la classe minoritaire par interpolation entre exemples existants).
  • ⚡ Piège : optimiser le rappel au détriment de la précision bloquerait des dizaines de milliers d'appels légitimes par jour — désastreux pour l'opérateur. La contrainte métier (coût des faux positifs élevé) impose de prioriser la précision.

Exercice 2 — Débogage de pipeline

Le code suivant contient 3 erreurs classiques. Identifie-les, explique pourquoi chacune est un problème, et propose la correction.

import pandas as pd
import numpy as np
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler, OrdinalEncoder
from sklearn.impute import SimpleImputer
from sklearn.model_selection import train_test_split, cross_val_score
import xgboost as xgb

# Dataset : prédire le défaut de crédit
df = pd.read_csv('credit_data.csv')
X = df.drop('defaut', axis=1)
y = df['defaut']

features_numeriques   = ['age', 'revenu', 'anciennete']
features_categoriques = ['region', 'type_emploi', 'niveau_etudes']

# Pipeline
preprocesseur = Pipeline([
    ('scaler', StandardScaler()),         # Erreur 1
    ('encoder', OrdinalEncoder()),
    ('imputer', SimpleImputer(strategy='mean'))
])

# Entraînement
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

preprocesseur.fit(X_train)
X_train_prep = preprocesseur.transform(X_train)
X_test_prep = preprocesseur.transform(X_test)

model = xgb.XGBClassifier(n_estimators=500, learning_rate=0.05)
model.fit(X_train_prep, y_train)

# Évaluation — Erreur 2
scores = cross_val_score(model, X_train_prep, y_train, cv=5, scoring='accuracy')
print(f"Score moyen : {np.mean(scores):.3f}")

# Prédiction sur nouvelles données — Erreur 3
nouveau_client = pd.DataFrame([{
    'age': 32, 'revenu': 2500, 'anciennete': 2,
    'region': 'Bretagne', 'type_emploi': 'Salarié', 'niveau_etudes': 'Bac+5'
}])
preprocesseur_nouveau = Pipeline([
    ('scaler', StandardScaler()),
    ('encoder', OrdinalEncoder()),
    ('imputer', SimpleImputer(strategy='mean'))
])
preprocesseur_nouveau.fit(nouveau_client)  # Erreur 3
X_nouveau_prep = preprocesseur_nouveau.transform(nouveau_client)
proba = model.predict_proba(X_nouveau_prep)[0, 1]
print(f"Probabilité de défaut : {proba:.3f}")

Corrections commentées

Erreur 1 — Ordre des étapes dans le pipeline

# Problème : StandardScaler est appliqué AVANT l'imputation et l'encodage
preprocesseur = Pipeline([
    ('scaler', StandardScaler()),     # ← plante sur les valeurs NaN et les catégories textuelles
    ('encoder', OrdinalEncoder()),
    ('imputer', SimpleImputer(strategy='mean'))
])

L'ordre doit être : imputer d'abord (combler les NaN), encoder ensuite (convertir le texte en nombres), scaler en dernier (normaliser les nombres). StandardScaler avant imputation plante car il ne peut pas calculer une moyenne sur des NaN. StandardScaler avant encodage plante car il ne peut pas normaliser du texte.

De plus, ce pipeline applique le même traitement à toutes les colonnes — les catégories ne doivent pas être normalisées avec StandardScaler.

Correction :

from sklearn.compose import ColumnTransformer

preprocesseur_num = Pipeline([
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler', StandardScaler())
])

preprocesseur_cat = Pipeline([
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('encoder', OneHotEncoder(handle_unknown='ignore'))  # OneHotEncoder, pas OrdinalEncoder pour région/type_emploi
])

preprocesseur = ColumnTransformer([
    ('num', preprocesseur_num, features_numeriques),
    ('cat', preprocesseur_cat, features_categoriques)
])

Erreur 2 — Cross-validation sur données déjà transformées, avec la mauvaise métrique

# Problème 1 : on fait la cross-validation sur X_train_prep (déjà transformé)
# alors que le preprocesseur a été fit uniquement sur X_train.
# En cross-validation, chaque fold doit re-fitter le preprocesseur sur son propre train
# pour éviter la fuite d'information (data leakage entre folds).

# Problème 2 : scoring='accuracy' sur un problème de crédit (classes déséquilibrées)
scores = cross_val_score(model, X_train_prep, y_train, cv=5, scoring='accuracy')

La cross-validation doit être faite sur le pipeline complet (preprocessing + modèle), pas sur les données déjà transformées. En transformant Xtrain avant la cross-validation, les statistiques (médiane, écart-type) ont été calculées sur tout Xtrain, y compris les données qui serviront de "test" dans chaque fold — c'est du data leakage.

Correction :

pipeline_complet = Pipeline([
    ('preprocessing', preprocesseur),
    ('modele', xgb.XGBClassifier(n_estimators=500, learning_rate=0.05))
])

# La cross-validation re-fit le preprocesseur à chaque fold — pas de leakage
scores = cross_val_score(pipeline_complet, X_train, y_train, cv=5, scoring='roc_auc')
print(f"AUC moyen : {np.mean(scores):.3f}")

Erreur 3 — Recréation et re-fit du preprocesseur sur les nouvelles données

# Problème : on crée un nouveau preprocesseur et on le fit sur nouveau_client
preprocesseur_nouveau = Pipeline([...])
preprocesseur_nouveau.fit(nouveau_client)  # ← catastrophique

Le preprocesseur doit être le même objet que celui fité sur X_train. Refitter sur une seule ligne calcule une médiane sur un seul exemple (inutile), une normalisation sur un seul exemple (l'écart-type vaut 0 → division par zéro). Les transformations appliquées seront totalement différentes de celles appliquées pendant l'entraînement → le modèle reçoit des données dans un format qu'il n'a jamais vu.

Correction :

# Le pipeline_complet encapsule déjà le preprocesseur fité sur X_train
# Il suffit d'appeler predict_proba directement
proba = pipeline_complet.predict_proba(nouveau_client)[0, 1]
print(f"Probabilité de défaut : {proba:.3f}")

Exercice 3 — Interprétation de métriques

Un data scientist livre les résultats suivants d'un modèle de détection de fraudes bancaires (1.5% des transactions sont des fraudes dans le dataset) :

{ } Code
=== Rapport d'évaluation ===
Accuracy   : 0.987
AUC-ROC    : 0.731
F1-score   : 0.421

              precision    recall  f1-score   support
    Normal       0.99      1.00      0.99     98500
    Fraude       0.72      0.27      0.39      1500

=== Cross-validation (5 folds) ===
AUC scores : [0.729, 0.733, 0.728, 0.736, 0.727]
Moyenne    : 0.731
Std        : 0.003

=== Feature importances (top 5) ===
montant_transaction         : 0.312
heure_transaction           : 0.187
pays_destination            : 0.156
frequence_7j                : 0.098
nb_transactions_meme_compte : 0.071

Réponds aux questions suivantes :

  1. Le data scientist présente les résultats au client en mettant en avant l'accuracy de 98.7%. Est-ce honnête ? Que lui réponds-tu ?

  2. Le rappel est à 0.27. Que signifie concrètement ce chiffre pour la banque ?

  3. L'AUC de 0.731 est-elle satisfaisante pour ce type de problème ? Que suggères-tu pour l'améliorer ?

  4. L'écart-type de la cross-validation est 0.003. Que t'indique cette valeur ?

  5. La feature montant_transaction a une importance de 0.312. Est-ce une information suffisante pour expliquer un refus à un client ?

Réponses commentées

Question 1 — L'accuracy de 98.7% est-elle honnête ?

Non. C'est le paradoxe classique des classes déséquilibrées. 98.5% des transactions sont normales — un modèle qui dit "normal" à chaque fois obtient déjà 98.5% d'accuracy sans détecter une seule fraude. L'accuracy de 98.7% ne prouve qu'une chose : le modèle dit "normal" presque tout le temps.

Ce qu'il faut dire au client : "L'accuracy est à 98.7% parce que 98.5% des transactions sont normales. La métrique pertinente est le rappel : notre modèle ne détecte que 27% des fraudes réelles. Sur 1 500 fraudes dans votre dataset de test, il en a manqué 1 095."

Question 2 — Le rappel à 0.27 concrètement

Sur 1 500 vraies fraudes dans les données de test, le modèle en a détecté 27% = 405 fraudes détectées. Il en a manqué 1 095 (faux négatifs). Pour une banque, chaque fraude manquée est une perte financière directe. Un rappel de 0.27 est très insuffisant pour un système de détection de fraude opérationnel — une cible raisonnable serait 0.70+.

Question 3 — L'AUC de 0.731 : satisfaisante ?

Passable mais insuffisante pour un déploiement. Un score de 0.731 signifie que le modèle classe correctement 73.1% des paires (une vraie fraude, une transaction normale) — c'est mieux que le hasard (0.5) mais un bon modèle de détection de fraude devrait atteindre 0.85+.

Pistes d'amélioration : - Rééquilibrer les classes : ajouter scale_pos_weight (paramètre XGBoost qui dit au modèle de donner plus de poids aux exemples positifs — fraudes — pendant l'entraînement ; typiquement réglé au ratio négatifs/positifs) dans XGBoost (ratio négatifs/positifs = 98.5/1.5 ≈ 66) ou utiliser SMOTE pour sur-échantillonner les fraudes. - Enrichir les features : la fréquence sur 7 jours est une bonne idée — ajouter aussi des features de vélocité (montant inhabituel pour ce compte spécifiquement). - Ajuster le seuil de décision : un modèle de classification retourne une probabilité (0.3, 0.7…) ; par défaut on dit "fraude si > 0.5". Abaisser ce seuil produit plus d'alertes (rappel élevé), le relever produit moins d'alertes mais plus fiables (précision élevée). Chercher le seuil qui maximise le rappel tout en maintenant une précision acceptable.

Question 4 — L'écart-type de 0.003

Très faible variance entre les folds — le modèle est stable. Il ne performe pas bien sur un fold particulier et mal sur d'autres. C'est une bonne nouvelle sur la robustesse : le problème n'est pas l'instabilité mais le niveau de performance général (AUC 0.731 trop bas).

Question 5 — L'importance de montant_transaction à 0.312 suffit-elle ?

Non. L'importance dans XGBoost mesure combien cette feature est utilisée pour construire les arbres — pas son impact sur une décision individuelle. Deux problèmes :

  1. Une importance élevée peut signifier que la feature est utilisée souvent pour des divisions mineures, pas qu'elle explique les grosses décisions.
  2. On ne sait pas si un montant élevé augmente ou diminue le score de fraude — ni de combien.

Pour expliquer un refus à un client, il faut les SHAP values : elles donnent la contribution de chaque feature pour chaque prédiction individuelle ("votre transaction de 4 500€ à 3h du matin vers un pays à risque a ajouté +0.43 au score de fraude").

import shap
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_test_prep)
shap.plots.waterfall(shap_values[exemple_fraude])
