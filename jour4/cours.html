21. Ce qu'est vraiment un réseau de neurones

Définition : le neurone artificiel

Un neurone artificiel est une fonction mathématique simple : il prend plusieurs entrées numériques, les multiplie chacune par un poids (un coefficient d'importance), additionne tout, puis applique une fonction d'activation non-linéaire pour décider s'il "s'active" ou non.

Formellement : sortie = activation(w1×x1 + w2×x2 + ... + wn×xn + biais)

  • Les entrées (x1, x2…) sont les données brutes ou la sortie des neurones précédents.
  • Les poids (w1, w2…) sont ce que le modèle apprend : ils encodent l'importance de chaque entrée.
  • Le biais est un terme constant qui déplace la fonction.
  • La fonction d'activation (ex : ReLU : max(0, x)) introduit la non-linéarité. Sans elle, empiler des couches ne servirait à rien — on obtiendrait juste une régression linéaire plus complexe.

Pourquoi plusieurs couches ?

Un seul neurone ne peut tracer qu'une frontière linéaire dans les données. On empile des couches pour apprendre des représentations de plus en plus abstraites :

{ } Code
Image (pixels bruts)
    → Couche 1 : contours, bords
    → Couche 2 : textures, coins
    → Couche 3 : formes (cercle, rectangle)
    → Couche 4 : parties d'objets (roue, oreille)
    → Couche finale : "chien" / "chat" / "voiture"

Chaque couche transforme les représentations de la couche précédente en quelque chose de plus abstrait et de plus utile pour la tâche finale. C'est pour ça qu'on appelle ça le deep learning : la profondeur (le nombre de couches) crée la richesse des représentations.

Backpropagation — l'intuition sans les maths

Le réseau fait une prédiction. Il se trompe. Comment apprend-il ?

Intuition : imaginez un GPS qui recalcule. Quand vous ratez un virage (erreur), le GPS ne recalcule pas toute la carte — il remonte uniquement les décisions qui ont mené à cette route. La backpropagation fait pareil : elle calcule, pour chaque poids, "de combien cet-te erreur est-ce la faute de ce poids ?", puis ajuste légèrement les poids dans la bonne direction.

Le réglage se fait via le gradient (la pente de l'erreur par rapport à chaque poids) et un learning rate (taux d'apprentissage) qui contrôle la taille des pas d'ajustement. Un learning rate trop grand → instabilité. Trop petit → apprentissage infiniment lent.

On répète ce cycle sur des batches (sous-ensembles des données) pendant plusieurs epochs (passages complets sur tout le dataset).

  • Batch : un sous-ensemble de données traité ensemble avant une mise à jour des poids. Taille typique : 32 à 256 exemples.
  • Epoch : un passage complet sur l'ensemble du dataset d'entraînement.
  • Learning rate : la taille des pas lors de l'ajustement des poids. Valeur typique : 0.001.

Code PyTorch — réseau simple

import torch
import torch.nn as nn

# Un réseau à 2 couches cachées pour classifier des données tabulaires
class ReseauSimple(nn.Module):
    def __init__(self, nb_features):
        super().__init__()
        self.couches = nn.Sequential(
            nn.Linear(nb_features, 64),  # couche 1 : nb_features → 64 neurones
            nn.ReLU(),                    # activation non-linéaire
            nn.Dropout(0.3),             # dropout : désactive 30% des neurones aléatoirement
            nn.Linear(64, 32),           # couche 2 : 64 → 32 neurones
            nn.ReLU(),
            nn.Linear(32, 1),            # couche sortie : 32 → 1 (classification binaire)
            nn.Sigmoid()                 # sortie entre 0 et 1
        )

    def forward(self, x):
        return self.couches(x)

# Entraînement minimal
modele = ReseauSimple(nb_features=20)
optimiseur = torch.optim.Adam(modele.parameters(), lr=0.001)
critere = nn.BCELoss()  # Binary Cross Entropy pour la classification binaire

for epoch in range(50):
    predictions = modele(X_train)
    perte = critere(predictions, y_train)
    optimiseur.zero_grad()
    perte.backward()      # backpropagation : calcul des gradients
    optimiseur.step()     # mise à jour des poids

Dropout : technique de régularisation. À chaque passage d'entraînement, on "éteint" aléatoirement une fraction des neurones. Cela force le réseau à ne pas sur-spécialiser — chaque partie du réseau doit être capable de travailler indépendamment. Résultat : meilleure généralisation sur de nouvelles données.

32. Quand le deep learning s'impose — et quand il ne s'impose PAS

La règle d'or

Le deep learning est puissant mais coûteux. Avant de l'utiliser, posez ces quatre questions : 1. Les données sont-elles en format non-structuré (image, texte, audio, vidéo) ? 2. Avez-vous plus de 100 000 exemples étiquetés ? 3. Avez-vous accès à un GPU et du temps de débogage ? 4. L'opacité du modèle est-elle acceptable pour ce contexte métier ?

Si la réponse à l'une de ces questions est "non", reconsidérez.

Le tableau décisionnel complet

Type de donnée Architecture recommandée Raison
Image CNN (ResNet, EfficientNet) Les convolutions capturent les patterns spatiaux locaux
Texte Transformer (BERT, CamemBERT) L'attention capte les dépendances longue distance
Série temporelle longue LSTM ou Temporal Fusion Transformer Mémoire sur les séquences
Données tabulaires XGBoost / LightGBM Plus performant, interprétable, rapide
Audio CNN sur spectrogramme ou Wav2Vec Conversion signal → représentation 2D
Graphe GNN (Graph Neural Network) Structure non-euclidienne

Pourquoi XGBoost bat le deep learning sur les tables structurées

C'est contre-intuitif, mais c'est un fait démontré par des dizaines de compétitions Kaggle et de publications académiques. Voici pourquoi :

  • Les données tabulaires ont peu de structure locale exploitable. CNN et Transformers tirent parti de la structure spatiale (images) ou séquentielle (texte). Une ligne d'un CSV n'a pas de telle structure — l'ordre des colonnes ne porte pas de sens.
  • XGBoost est invariant aux features non-informatives. Un réseau de neurones doit apprendre à les ignorer. XGBoost les écarte naturellement via le gain d'information.
  • Le rapport données/paramètres est défavorable. Un réseau de neurones a des millions de paramètres. Sur 50 000 lignes et 30 colonnes, il y a trop peu de signal pour les contraindre efficacement.
  • Les features engineered sont critiques. L'expert métier sait que "CA mensuel / nombre de clients" est une feature utile. Les arbres l'exploitent directement. Les réseaux de neurones doivent la redécouvrir.
# Benchmark concret sur données tabulaires (UCI Adult Dataset, 48K lignes)
# XGBoost vs MLP (réseau de neurones multicouche)

from xgboost import XGBClassifier
from sklearn.neural_network import MLPClassifier
from sklearn.metrics import accuracy_score

# XGBoost : ~87% accuracy, 0.3s d'entraînement
xgb = XGBClassifier(n_estimators=300, max_depth=6)
xgb.fit(X_train, y_train)  # 0.3 secondes

# MLP : ~85% accuracy, 45s d'entraînement (CPU)
mlp = MLPClassifier(hidden_layer_sizes=(256, 128, 64), max_iter=200)
mlp.fit(X_train, y_train)  # 45 secondes
# XGBoost : plus rapide, plus précis, interprétable avec SHAP

Le coût réel du deep learning

  • Données : minimum 10 000 exemples pour des résultats raisonnables, 100 000+ pour des résultats robustes. Sur de l'image médicale annotée par des experts : comptez 50€ à 500€ par image.
  • Calcul : entraîner ResNet50 from scratch sur ImageNet prend 90 heures sur 8 GPU V100. Louer ces GPU : ~2 000 €.
  • Débogage : un réseau qui ne converge pas peut passer des jours à diagnostiquer. Les hyperparamètres (learning rate, architecture, augmentation de données) interagissent de façon imprévisible.
  • Opacité : un réseau de neurones est une boite noire. Dans les secteurs réglementés (crédit, santé, assurance), l'obligation d'expliquabilité peut interdire son usage.

:::situation Lactalis — détection de défauts qualité sur images

Lactalis, groupe laitier (Président, Galbani), contrôle la qualité des emballages en sortie de ligne. Le problème : détecter les défauts (étiquette mal collée, emballage déchiré, date d'expiration illisible) sur 600 emballages par minute, 24h/24.

Un système de règles basé sur des seuils de couleur et de contraste avait un taux de faux positifs de 12% — trop de produits sains écartés. Un CNN EfficientNet entraîné sur 80 000 images annotées (3 semaines de collecte) descend à 0.8% de faux positifs, et détecte des défauts que les règles ne voyaient pas (microfissures, variations de teinte subtiles).

La décision de basculer : type de données (image) + volume (80K exemples) + ROI quantifiable (12% → 0.8% de rejet abusif sur 600 emballages/minute = économies mesurables) → deep learning justifié. :::

43. Transfer Learning : ne pas réinventer la roue

Définition

Le transfer learning (apprentissage par transfert) consiste à prendre un modèle déjà entraîné sur une grande tâche générique, et à réutiliser les représentations qu'il a apprises pour résoudre une tâche spécifique — souvent avec beaucoup moins de données.

L'idée centrale : un modèle entraîné sur 1,2 million d'images (ImageNet) a développé dans ses premières couches une capacité générique à détecter des contours, des textures, des formes. Ces capacités sont transférables à n'importe quelle tâche vision, qu'il s'agisse de radiographies, de photos satellites ou d'images industrielles.

La métaphore du médecin

Un médecin généraliste qui souhaite devenir cardiologue n'efface pas tout ce qu'il a appris. Il réutilise ses connaissances en anatomie, physiopathologie, pharmacologie, et y ajoute une spécialisation. Cela lui prend 3 ans, pas 10.

Un étudiant qui partirait de zéro directement en cardiologie devrait tout réapprendre : 10 ans, et avec une formation bien moins solide sur les bases générales.

C'est exactement le principe du transfer learning : réutiliser une base solide plutôt que repartir de zéro.

Les deux stratégies

Feature extraction (geler les poids)

On utilise le modèle pré-entraîné comme un extracteur de features fixe. On gèle tous ses poids (on ne les modifie pas pendant l'entraînement). On ajoute uniquement une ou deux couches finales que l'on entraîne sur les nouvelles données.

Quand l'utiliser : tâche similaire à la tâche source, très peu de données (< 5 000 exemples), budget GPU limité.

Fine-tuning (réentraîner les dernières couches)

On commence avec les poids pré-entraînés, mais on laisse les dernières couches du modèle se réajuster sur les nouvelles données. Les premières couches restent gelées (représentations génériques), les dernières couches se spécialisent.

Quand l'utiliser : tâche différente de la tâche source, données suffisantes (5 000 à 50 000 exemples), GPU disponible.

import torchvision.models as models
import torch.nn as nn

# AVANT : entraîner from scratch (impossible avec 5000 images)
# class CNN(nn.Module):
#     def __init__(self):
#         super().__init__()
#         # Des dizaines de couches à initialiser aléatoirement...
#         # 25M paramètres à apprendre depuis zéro
#         # Nécessite 1M+ d'images pour converger

# APRÈS : Transfer Learning avec ResNet50 pré-entraîné
modele = models.resnet50(pretrained=True)  # 25M paramètres pré-entraînés sur ImageNet

# Stratégie 1 — Feature extraction : geler tous les poids
for param in modele.parameters():
    param.requires_grad = False  # aucun poids modifié sauf la tête

# Remplacer la dernière couche par notre classifieur (2 classes : sain / cancéreux)
nb_features = modele.fc.in_features
modele.fc = nn.Linear(nb_features, 2)  # seule cette couche est entraînée

# Stratégie 2 — Fine-tuning : réentraîner les dernières couches
for param in modele.layer4.parameters():  # débloquer la dernière "étape" de ResNet
    param.requires_grad = True
modele.fc = nn.Linear(nb_features, 2)  # nouvelle tête

# Entraînement : seulement quelques epochs suffisent
optimiseur = torch.optim.Adam([
    {'params': modele.layer4.parameters(), 'lr': 1e-4},  # petite LR pour les couches gelées
    {'params': modele.fc.parameters(),     'lr': 1e-3},  # grande LR pour la nouvelle tête
])

:::situation Sanofi — classification de cellules cancéreuses avec 5 000 images

Sanofi développe un test diagnostique basé sur l'analyse d'images de cellules. Le dataset disponible : 5 000 images annotées par des pathologistes (coût : ~200€/image → 1M€ de budget annotation). 54. HuggingFace en profondeur : du modèle à la production

Ce qu'est HuggingFace

HuggingFace est une plateforme et une bibliothèque Python qui centralise les modèles de deep learning pré-entraînés, les datasets, et les outils pour les utiliser. En 2024 : 500 000+ modèles, 150 000+ datasets, 200 000+ Spaces (démos interactives).

La proposition de valeur : au lieu de coder une architecture Transformer from scratch (plusieurs semaines de travail), on télécharge un modèle pré-entraîné en une ligne, et on l'adapte à sa tâche.

Tokenisation — pourquoi on ne passe pas du texte brut à un modèle

Un modèle de language travaille sur des nombres, pas sur des caractères. Il faut d'abord tokeniser le texte : le découper en unités (tokens) et associer chaque token à un identifiant numérique.

Un token n'est pas forcément un mot entier. Les modèles modernes utilisent des algorithmes (BPE, WordPiece) qui découpent les mots rares en sous-unités. Ainsi "biotechnologie" peut devenir ["bio", "##techno", "##logie"] selon le vocabulaire du modèle.

from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("camembert-base")

texte = "Le client est très satisfait de la livraison."
tokens = tokenizer(texte, return_tensors="pt", padding=True, truncation=True)

print(tokens['input_ids'])
# tensor([[5, 72, 3421, 18, 943, 8921, 16, 72, 9823, 6]])
# Chaque nombre = un token dans le vocabulaire du modèle (32 000 tokens pour CamemBERT)

Les Pipelines HuggingFace — utilisation immédiate

from transformers import pipeline

# Classification de sentiment (sans entraînement)

# [{'label': 'POSITIVE', 'score': 0.98}]

Fine-tuner CamemBERT sur des avis clients en français

CamemBERT est le BERT français : un Transformer entraîné sur 138 Go de texte français (Wikipedia, CommonCrawl, livres). Il comprend la grammaire, le vocabulaire et les nuances du français.

from transformers import AutoTokenizer, AutoModelForSequenceClassification, Trainer, TrainingArguments
from datasets import Dataset
import pandas as pd

# Données : avis clients en français avec labels (0=négatif, 1=positif)
df = pd.read_csv("avis_clients.csv")  # colonnes : "texte", "label"
dataset = Dataset.from_pandas(df).train_test_split(test_size=0.2)

# AVANT : entraîner from scratch (impraticable sans corpus massif)
# model = BertModel(config)  # 110M paramètres initialisés aléatoirement

# APRÈS : partir de CamemBERT pré-entraîné
tokenizer = AutoTokenizer.from_pretrained("camembert-base")
modele = AutoModelForSequenceClassification.from_pretrained(
    "camembert-base",
    num_labels=2  # positif / négatif
)

def tokeniser(exemples):
    return tokenizer(exemples["texte"], truncation=True, padding="max_length", max_length=128)

dataset_tokenise = dataset.map(tokeniser, batched=True)

# Configuration de l'entraînement
args = TrainingArguments(
    output_dir="./modele_avis",
    num_train_epochs=3,          # 3 epochs suffisent avec transfer learning
    per_device_train_batch_size=16,
    learning_rate=2e-5,          # learning rate faible : on ajuste légèrement des poids déjà bons
    evaluation_strategy="epoch",
    save_strategy="epoch",
    load_best_model_at_end=True,
)

trainer = Trainer(
    model=modele,
    args=args,
    train_dataset=dataset_tokenise["train"],
    eval_dataset=dataset_tokenise["test"],
)

trainer.train()  # 3-5 minutes sur GPU, 30-45 minutes sur CPU

# Inférence
inputs = tokenizer("La livraison était en retard et l'emballage abimé", return_tensors="pt")
with torch.no_grad():
    logits = modele(**inputs).logits
prediction = logits.argmax(-1).item()  # 0 = négatif

65. Les architectures clés à connaître sans les coder

CNN — Réseaux de neurones convolutifs

Définition : Un CNN applique des filtres (petites matrices de poids) qui glissent sur l'image. Chaque filtre détecte un pattern local (un contour horizontal, une couleur particulière). Le pooling réduit ensuite la résolution pour garder l'essentiel. En empilant des convolutions + poolings, on passe des pixels bruts à des représentations de plus en plus abstraites.

L'avantage clé : les filtres sont partagés sur toute l'image. Un filtre qui détecte un bord oblique en haut à gauche fonctionne aussi en bas à droite. C'est ce qu'on appelle l'invariance par translation — propriété naturelle pour les images.

Cas réel français : PSA Stellantis utilise des CNN pour inspecter les carrosseries en sortie d'usine à Sochaux. 98% des micro-rayures détectées vs 73% par inspection humaine. Cadence : une inspection complète en 2 secondes.

LSTM / GRU — Mémoire à long terme

Définition : Un LSTM (Long Short-Term Memory) est un réseau récurrent qui maintient une mémoire interne entre les étapes de la séquence. À chaque pas, il décide quoi mémoriser, quoi oublier, et quoi laisser passer. Cela lui permet de capturer des dépendances à longue distance dans une séquence.

Le GRU (Gated Recurrent Unit) est une version simplifiée du LSTM, plus rapide à entraîner, souvent aussi performant.

Cas réel français : EDF utilise des LSTM pour prédire la consommation électrique nationale à 72h. Le modèle prend en entrée les 168 dernières heures de consommation (une semaine), la météo, le calendrier (jours fériés, vacances scolaires) et prédit heure par heure les 72 suivantes. Précision : ±1.2% vs ±3.8% pour les méthodes ARIMA classiques.

Transformer & Attention — Pourquoi tout le monde a abandonné les LSTM pour le texte

Définition : Le mécanisme d'attention permet à chaque mot d'une phrase de "regarder" tous les autres mots simultanément pour calculer sa représentation contextuelle. L'architecture Transformer (2017, Google) remplace la récurrence par l'attention pure.

Avantage sur les LSTM : les LSTM traitent les tokens un par un, séquentiellement. Le Transformer traite tous les tokens en parallèle → vitesse d'entraînement 10× à 100× supérieure. De plus, l'attention directe entre tokens distants résout le problème de la mémoire à longue distance que les LSTM peinent à gérer.

Cas réel français : Doctolib utilise des Transformers pour analyser les notes cliniques textuelles (en français) et suggérer des codes CIM-10 aux médecins lors de la facturation. Gain de temps estimé : 8 minutes par consultation.

Diffusion Models — La logique derrière Stable Diffusion

Définition : Un modèle de diffusion apprend à inverser un processus de dégradation progressive. Pendant l'entraînement : on prend une image réelle, on ajoute du bruit gaussien étape par étape jusqu'à obtenir du bruit pur. Le modèle apprend à désbruiter à chaque étape. Lors de la génération : on part de bruit pur et on applique le processus inverse, guidé par un texte (CLIP encoder).

Intuition : le modèle a appris la "physique" de toutes les images du monde. Quand on lui donne du bruit et une description, il sculpte progressivement l'image en partant de ce bruit.

Cas réel : L'Oréal utilise Stable Diffusion fine-tuné pour générer des visuels de campagne publicitaire en cours de créa, permettant aux équipes créatives de tester 50 directions artistiques en 2 heures au lieu de 3 semaines.

Les LSTM traitent les tokens séquentiellement (lent). Les Transformers traitent tous les tokens en parallèle (10-100× plus rapide) et gèrent mieux les dépendances longue distance via l'attention directe entre tokens distants.

76. Déployer un modèle PyTorch : de l'expérience à la production

Le gap notebook → production

Un modèle qui "fonctionne" en notebook Jupyter n'est pas un modèle en production. Les problèmes typiques :

  • Le notebook charge les données depuis un fichier local, pas une API
  • Le modèle tient en mémoire d'une session Python mais pas de façon persistante
  • Les temps d'inférence ne sont pas mesurés
  • Il n'y a pas de gestion des erreurs ni de monitoring
  • L'environnement de dépendances n'est pas fixé (versions PyTorch, CUDA…)

Sérialisation — sauvegarder le modèle

Définition : La sérialisation consiste à convertir un objet en mémoire (le modèle et ses poids) en un fichier persistant sur disque.

import torch

# Option 1 — torch.save : format natif PyTorch
torch.save(modele.state_dict(), "modele_poids.pt")  # sauvegarde les poids uniquement
modele_charge = ReseauSimple(nb_features=20)
modele_charge.load_state_dict(torch.load("modele_poids.pt"))

# Option 2 — ONNX : format interopérable (peut tourner sans PyTorch)
import torch.onnx
dummy_input = torch.randn(1, 20)  # exemple d'entrée de même forme que les vraies données
torch.onnx.export(modele, dummy_input, "modele.onnx")
# → Le fichier .onnx tourne sur ONNX Runtime, TensorRT, CoreML, etc.
# → Indépendant de PyTorch : utilisable en Java, C++, dans un navigateur (WebAssembly)

Quand utiliser ONNX : déploiement sur mobile, edge computing, environnements sans Python, ou quand on veut s'affranchir de la version PyTorch.

Optimisation : quantization et pruning

Quantization : réduire la précision numérique des poids du modèle. Au lieu de stocker chaque poids en float32 (32 bits), on utilise int8 (8 bits). Résultat : modèle 4× plus léger, inférence 2-3× plus rapide sur CPU, perte de précision < 1% sur la plupart des tâches.

Pruning : supprimer les connexions (poids proches de zéro) qui contribuent peu à la sortie. Un réseau entraîné est typiquement sur-paramétré. On peut souvent supprimer 70 à 90% des connexions sans perte significative de performance.

# Quantization dynamique (la plus simple, pas besoin de données de calibration)
import torch.quantization

modele_quantifie = torch.quantization.quantize_dynamic(
    modele,
    {nn.Linear},  # quantifier les couches linéaires
    dtype=torch.qint8
)

# Comparaison de taille
import os
torch.save(modele.state_dict(), "modele_fp32.pt")
torch.save(modele_quantifie.state_dict(), "modele_int8.pt")
print(f"FP32: {os.path.getsize('modele_fp32.pt') / 1e6:.1f} MB")
print(f"INT8: {os.path.getsize('modele_int8.pt') / 1e6:.1f} MB")
# FP32: 88.3 MB
# INT8: 22.1 MB  ← 4× plus léger

Exposer le modèle avec FastAPI

from fastapi import FastAPI
from pydantic import BaseModel
import torch, numpy as np

app = FastAPI()

# Chargement du modèle au démarrage (pas à chaque requête)
modele = ReseauSimple(nb_features=20)
modele.load_state_dict(torch.load("modele_poids.pt", map_location="cpu"))
modele.eval()  # mode inférence : désactive le dropout

class RequeteInference(BaseModel):
    features: list[float]  # liste de 20 features

@app.post("/predict")
def predire(requete: RequeteInference):
    with torch.no_grad():
        x = torch.tensor([requete.features], dtype=torch.float32)
        proba = modele(x).item()
    return {"probabilite": proba, "prediction": int(proba > 0.5)}

Flashcards

Quelle est la différence entre torch.save et ONNX pour la sérialisation d'un modèle ?

Voir la réponse
Réponse

`torch.save` est le format natif PyTorch, simple mais dépendant de PyTorch. ONNX est interopérable : le fichier tourne sur ONNX Runtime, TensorRT, CoreML — sans PyTorch, en Java, C++, mobile.

Qu'est-ce que la quantization et quel gain typique apporte-t-elle ?

Voir la réponse
Réponse

Réduire la précision des poids de float32 à int8. Résultat : modèle 4× plus léger, inférence 2-3× plus rapide sur CPU, perte de précision < 1%.

Pourquoi appeler modele.eval() avant l'inférence ?

Voir la réponse
Réponse

Pour désactiver le Dropout (qui est aléatoire et ne doit pas s'appliquer en production) et la Batch Normalization en mode entraînement. Sans eval(), les prédictions seraient non-déterministes.

Quel est le principal avantage de charger le modèle au démarrage du serveur plutôt qu'à chaque requête ?

Voir la réponse
Réponse

Charger un modèle prend 2 à 10 secondes. Faire ça à chaque requête rendrait l'API inutilisable. On charge une fois en mémoire et on réutilise pour toutes les requêtes.
