Type : ML classique supervisé (classification binaire)

Justification : Labels disponibles (fraude/normale), données tabulaires (montant, heure, pays, etc.), volume gérable pour sklearn/XGBoost. La contrainte d'explicabilité AMF oriente vers un modèle interprétable ou SHAP values.

Type : ML non-supervisé (clustering)

Justification : Pas de label "type de client" → on ne peut pas faire du supervisé. L'objectif est de découvrir des groupes naturels.

Bibliothèques : scikit-learn (K-Means, DBSCAN) + pandas (la bibliothèque standard pour manipuler des tableaux de données en Python — lire un CSV, filtrer des lignes, normaliser des colonnes) pour le preprocessing. Ce que le ML ne fait PAS : Dire à la place du métier ce que signifient les clusters. Le data scientist propose des regroupements, le business les valide et les nomme.

Recommandation réaliste : Commencer par Option 1 (règles). Mesurer les cas non couverts. Si >30% de fallback → Option 2. GenAI n'est pas le bon outil pour une classification à 5 catégories fixes avec des données disponibles.

Exercice 3 — Cas piège (15 min)

Piège A

Un DSI vous dit : "On a des données sur nos 200 000 clients, on veut faire du clustering pour les segmenter, puis prédire qui va churner."

Où est le piège dans cette phrase ?

Réponse

Deux problèmes fondamentaux mélangés :

  1. Le clustering (non-supervisé) donne des segments, pas une prédiction de churn. Pour prédire le churn, il faut un modèle supervisé avec un label "a churné" dans les données historiques.

  2. "Faire du clustering puis prédire" semble logique mais crée un problème : les clusters ne sont pas des features valides pour prédire le churn. C'est une fuite de données potentielle (data leakage) si les clusters sont calculés sur l'ensemble des données.

La bonne architecture : - Clustering pour comprendre les profils clients → insights marketing - Churn prediction séparément : supervisé, avec label "actif/inactif à J+90"

Piège B

Un client vous montre son modèle de détection de fraude : accuracy = 99.5%. Il est ravi. Vous devez l'inquiéter. Pourquoi ?

Réponse

Paradoxe de l'accuracy sur données déséquilibrées

Si 0.5% des transactions sont frauduleuses, un modèle qui dit toujours "normal" a une accuracy de 99.5%. Il ne détecte aucune fraude.

La bonne métrique sur ce problème : - Précision : sur les alertes levées, combien sont vraies fraudes ? (éviter les faux positifs = bloquer des clients légitimes) - Rappel : sur les vraies fraudes, combien sont détectées ? (éviter les faux négatifs = laisser passer des fraudes) - F1-score ou AUC-ROC : balance les deux

La métrique à optimiser dépend du coût métier : est-il pire de rater une fraude ou de bloquer un bon client ?
