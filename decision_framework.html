JOUR 1 — Paysage ML : Quoi choisir et pourquoi

Décision 1.1 — Règles vs ML

Contexte Choisir
Logique métier explicite, stable, auditée (banque, médical réglementé) Règles
Données disponibles, patterns complexes, logique non-formulable ML
Règles connues mais exceptions nombreuses Règles + ML en fallback
⚡ Piège

Automatiquement sauter au ML parce que c'est "plus moderne". Un if/else bien pensé est souvent plus fiable, plus rapide, et moins coûteux.

Exemple réel : La Société Générale utilise des règles pour le scoring réglementaire Bâle III (auditabilité obligatoire), et du ML pour la détection de fraude (patterns trop complexes pour des règles).

Décision 1.2 — Type de ML

Question Réponse
J'ai des labels (y) → je veux prédire Supervisé
Pas de labels → je veux comprendre la structure Non-supervisé
Séquence de décisions, récompense différée RL (rare en prod client)
Texte, image, audio, vidéo Deep Learning (voir Jour 4)
Données tabulaires structurées ML classique d'abord (voir Jour 2)

✅ Règle d'or

Ne jamais sortir l'artillerie lourde (deep/GenAI) si une régression logistique ou un XGBoost fait le job. Coût × Maintenance × Explicabilité.

Décision 1.4 — Quelle bibliothèque Python

Besoin Bibliothèque Pourquoi pas l'autre
ML classique rapide, pipeline prod scikit-learn Référence universelle, tous les algos ont la même syntaxe
Gradient boosting sur données tabulaires XGBoost / LightGBM Sklearn trop lent sur gros volumes
Séries temporelles statistiques statsmodels / Prophet Sklearn n'a pas ces modèles
Deep Learning flexible, recherche PyTorch TF plus lourd à déboguer
Deep Learning prod, NLP state-of-art HuggingFace Transformers PyTorch seul = tout from scratch
Experiment tracking MLflow Sinon tu perds tes runs (voir Jour 5)
