21. Le problème que MLOps résout

Définition : MLOps (Machine Learning Operations) désigne l'ensemble des pratiques, outils et processus qui permettent de faire passer un modèle ML d'un notebook de développement à un système fiable et maintenable en production. C'est la jonction entre le travail du data scientist (construire un modèle) et celui de l'ingénieur (faire tourner un système en continu).

Sans MLOps, un modèle ML reste une expérience de laboratoire. Avec MLOps, il devient un service.

La vallée de la mort

Selon les études sectorielles, entre 70 et 80% des modèles ML développés ne passent jamais en production. Les raisons ne sont presque jamais techniques — le modèle fonctionne. Elles sont organisationnelles et opérationnelles :

Ce qui marche en notebook : ───────────────────────────────────────────────────── ✓ Les données sont propres (le data scientist les a nettoyées) ✓ L'environnement est fixe (Python 3.11, pandas 2.0, exact) ✓ Les prédictions sont rapides (200 lignes de test) ✓ Si ça plante, on relance manuellement Ce qui arrive en production : ───────────────────────────────────────────────────── ✗ Les données arrivent en flux, avec des formats inattendus ✗ Le serveur de prod a une version différente de numpy ✗ Le modèle doit répondre en 50ms sur 10 000 requêtes/heure ✗ Si ça plante à 3h du matin, personne ne le sait

Situation réelle : Société Générale, modèle de scoring "en production"

Une équipe data de la Société Générale a développé un excellent modèle de scoring de crédit. Il est "en production" depuis 8 mois. En réalité : chaque lundi matin, un data scientist exporte les prédictions du modèle dans un fichier Excel, qu'il envoie par email à l'équipe crédit, qui copie-colle les valeurs dans le système interne. Si le data scientist est en vacances, le scoring n'est pas fait. Si le format du fichier source change, le modèle plante silencieusement. C'est ce que MLOps est censé rendre impossible.

Le cycle de vie d'un modèle ML

{ } Code
1. DÉVELOPPEMENT
   Expérimentation → feature engineering → entraînement → évaluation
   Outils : Jupyter, sklearn, XGBoost, MLflow tracking

2. VALIDATION
   Tests de performance, tests de robustesse, validation métier
   Outils : pytest, MLflow, comparaison de versions

3. DÉPLOIEMENT
   Packager le modèle, l'exposer via une API, gérer les dépendances
   Outils : FastAPI, Docker, MLflow serving

4. MONITORING
   Surveiller les performances réelles, détecter la dérive
   Outils : Evidently, Prometheus, dashboards métier

5. RÉ-ENTRAÎNEMENT
   Déclencher automatiquement quand les performances chutent
   Outils : Airflow, Prefect, MLflow registry

:::callout-insight MLOps n'est pas une technologie — c'est une discipline. On peut faire du MLOps correct avec des outils simples (MLflow + FastAPI + un cron job), ou du MLOps défaillant avec une stack Kubernetes/Kubeflow mal configurée. Ce qui compte : la traçabilité, la reproductibilité, et l'automatisation des tâches répétitives. :::

Voir la réponse
Réponse

MLOps = pratiques + outils pour passer un modèle du notebook à la production. Ça existe parce que 70-80% des modèles ne passent jamais en prod — non par problème technique, mais par manque de traçabilité, de reproductibilité et d'automatisation.

Voir la réponse
Réponse

1. Les données réelles arrivent en flux avec des formats inattendus. 2. L'environnement de prod a des versions de bibliothèques différentes. 3. Pas de monitoring — le modèle peut planter ou dériver pendant des semaines sans que personne le sache.

Quelles sont les 5 étapes du cycle de vie d'un modèle ML ?

Voir la réponse
Réponse

1. Développement (expérimentation) → 2. Validation (tests) → 3. Déploiement (API + packaging) → 4. Monitoring (surveillance en prod) → 5. Ré-entraînement (automatisé sur dérive détectée).

32. MLflow en profondeur : tracking, registry, serving

Définition : MLflow est une plateforme open source de gestion du cycle de vie des modèles ML. Elle résout un problème universel : après des dizaines d'expériences, impossibilité de savoir quelle version du modèle est en production, avec quels paramètres, et pourquoi celle-là plutôt qu'une autre.

Situation réelle : Decathlon, modèle de prévision de stocks

Trois data scientists chez Decathlon travaillent en parallèle sur un modèle de prévision de ruptures de stock. Sans MLflow : chacun sauvegarde ses modèles dans des fichiers nommés model_v2_final_vraiment_final.pkl, les métriques dans des Excel différents, et personne ne sait quelle version tourne en production depuis 2 mois. Avec MLflow : chaque expérience est loguée automatiquement, comparable en un clic, et le modèle en production est traçable jusqu'au commit Git qui l'a produit.

Les 4 composantes de MLflow

1. MLflow Tracking — le carnet de laboratoire automatique

import mlflow
import xgboost as xgb
from sklearn.metrics import accuracy_score

# Démarrer une expérience nommée (groupe logique de runs)
mlflow.set_experiment("decathlon-stock-forecast")

with mlflow.start_run(run_name="xgb-lr0.05-n500"):
    # Paramètres : les réglages utilisés
    params = {"n_estimators": 500, "learning_rate": 0.05, "max_depth": 6}
    mlflow.log_params(params)

    # Entraînement
    model = xgb.XGBClassifier(**params)
    model.fit(X_train, y_train)

2. MLflow Registry — versionning des modèles avec stages

Le Registry est une bibliothèque centrale pour les modèles validés. Chaque modèle a un stage : None (expérimental) → Staging (en test) → Production (actif) → Archived (retraité).

# Promouvoir un run vers le Registry
run_id = "abc123def456"  # l'ID du meilleur run dans l'UI MLflow
mlflow.register_model(
    model_uri=f"runs:/{run_id}/model",
    name="decathlon-stock-model"
)

# Passer en Staging pour les tests
client = mlflow.tracking.MlflowClient()
client.transition_model_version_stage(
    name="decathlon-stock-model",
    version=3,
    stage="Staging"
)

# Passer en Production après validation
client.transition_model_version_stage(
    name="decathlon-stock-model",
    version=3,
    stage="Production"
)

3. Charger le modèle en production

# En inférence : charger toujours la version "Production"
# Si on déploie une nouvelle version, elle passe d'abord en Staging
# → tests automatiques → puis Production. L'ancien passe en Archived.
model = mlflow.xgboost.load_model(
    model_uri="models:/decathlon-stock-model/Production"
)

# Prédiction directe — même interface que sklearn
predictions = model.predict(X_new)

Flashcards

Quelle est la différence entre MLflow Tracking et MLflow Registry ?

Voir la réponse
Réponse

Tracking = journal de toutes les expériences (params, métriques, modèles bruts). Registry = bibliothèque des modèles *validés*, avec versionning et stages (Staging/Production/Archived). On logue tout dans Tracking, on ne promeut dans Registry que ce qui est prêt à être déployé.

Pourquoi avoir des stages (Staging → Production → Archived) dans MLflow Registry ?

Voir la réponse
Réponse

Pour séparer les modèles en test de ceux en production. Un nouveau modèle passe d'abord en Staging (tests automatiques, validation métier). Seulement si les tests passent → Production. L'ancien modèle passe en Archived mais reste récupérable si on doit rollback.

Comment charger en production toujours la dernière version "Production" d'un modèle ?

Voir la réponse
Réponse

`mlflow.xgboost.load_model("models:/nom-modele/Production")` — MLflow résout automatiquement quelle version porte le stage "Production". Quand on promeut une nouvelle version, le code de prod n'a pas besoin d'être modifié.

43. Data drift et model drift : détecter la dégradation silencieuse

Définition du data drift : Le data drift se produit quand la distribution des données d'entrée change après le déploiement. Le modèle a appris sur des données de 2022 ; en 2024, les clients ont un profil différent, les produits ont changé, les comportements ont évolué. Le modèle n'a pas changé — mais les données qu'il reçoit ne ressemblent plus à celles sur lesquelles il a été entraîné.

Définition du model drift (ou concept drift) : Le model drift se produit quand la relation entre les données et la cible change. Exemple : en 2019, un client qui demande un crédit avec un CDI et un apport de 20% est un bon risque. En 2024, avec l'inflation et les taux d'intérêt à 4%, ce même profil peut être fragile. Le modèle prédit avec les mêmes features, mais la réalité sous-jacente a changé.

{ } Code
Data drift   → les ENTRÉES changent
               Ex : les clients sont en moyenne plus âgés qu'à l'entraînement
               → le modèle extrapole en dehors de sa zone d'apprentissage

Model drift  → la RELATION entrées/cible change
               Ex : ce qui prédisait le churn en 2022 ne prédit plus rien en 2024
               → le modèle a les bonnes données mais le mauvais modèle mental

Situation réelle : scoring crédit pré-COVID

Une banque régionale déploie en janvier 2020 un modèle de scoring crédit entraîné sur 2017-2019. En mars 2020, le COVID arrive. Les revenus chutent, les comportements changent radicalement, des catégories professionnelles entières basculent en risque élevé. Le modèle continue à prédire comme si on était en 2019 — il accepte des dossiers qui vont faire défaut, refuse des dossiers qui auraient été solides. Les métriques en production semblent stables pendant 6 semaines (le délai entre l'octroi et le premier défaut est de 2-3 mois). Quand la dégradation devient visible, il est trop tard.

Comment détecter le data drift avec Evidently

Evidently est une bibliothèque open source qui compare la distribution des données de référence (entraînement) avec les données actuelles (production).

import pandas as pd
from evidently.report import Report
from evidently.metric_preset import DataDriftPreset, TargetDriftPreset

# Données de référence = le jeu d'entraînement
reference_data = pd.read_csv("train_data_2022.csv")

# Données actuelles = ce qui arrive en production aujourd'hui
current_data = pd.read_csv("prod_data_2024_q1.csv")

# Rapport de drift
report = Report(metrics=[
    DataDriftPreset(),       # drift des features d'entrée
    TargetDriftPreset(),     # drift de la variable cible (si disponible)
])
report.run(reference_data=reference_data, current_data=current_data)
report.save_html("drift_report.html")

Comment Evidently mesure le drift : pour chaque feature, il applique un test statistique adapté à son type : - Variables continues : test de Kolmogorov-Smirnov (est-ce que les deux distributions viennent de la même loi ?) - Variables catégorielles : test du chi² (est-ce que les proportions ont changé ?) - PSI (Population Stability Index) : mesure l'ampleur du déplacement, avec des seuils pratiques (PSI < 0.1 = stable, 0.1-0.2 = surveillance, > 0.2 = action requise)

:::rule Règle des 3 niveaux de monitoring : 1. Technique : latence, taux d'erreur, disponibilité (à surveiller en temps réel) 2. Data : drift des features d'entrée (à surveiller hebdomadairement) 3. Métier : performance réelle sur les décisions prises (à surveiller mensuellement, délai incompressible) :::

Flashcards

Quelle est la différence entre data drift et model drift ?

Voir la réponse
Réponse

Data drift = les données d'entrée ont changé (profil client différent). Model drift = la *relation* entre les données et la cible a changé (ce qui prédisait le churn avant ne le prédit plus). Le premier se détecte statistiquement ; le second nécessite des labels récents pour être confirmé.

Pourquoi un modèle peut-il sembler stable pendant des semaines alors qu'il se dégrade ?

Voir la réponse
Réponse

Parce qu'il y a un délai entre la décision et le résultat observable. Pour un crédit, le défaut arrive 2-3 mois après l'octroi. Le modèle peut prendre de mauvaises décisions aujourd'hui, mais les métriques de performance ne le montreront que dans 2-3 mois.

Qu'est-ce que le PSI et à partir de quelle valeur agit-on ?

Voir la réponse
Réponse

PSI = Population Stability Index, mesure l'ampleur du déplacement d'une distribution. PSI < 0.1 = stable (rien à faire). 0.1-0.2 = surveillance accrue. > 0.2 = action requise (re-entraîner ou investiguer la cause).

54. Airflow / Prefect : orchestrer les pipelines

Définition : Un pipeline d'orchestration est un ensemble de tâches automatisées qui s'enchaînent selon un ordre défini et une cadence programmée. Ce n'est pas une pipeline sklearn (chaîne de transformations sur des données) — c'est un chef d'orchestre qui décide quand et dans quel ordre faire tourner des scripts.

La structure de base est un DAG (Directed Acyclic Graph — graphe orienté sans cycle) : chaque tâche a des dépendances (elle ne démarre que si ses prédécesseurs ont réussi), et l'ensemble forme un graphe sans boucle.

Exemple de DAG pour le ré-entraînement hebdomadaire : [ Extraire données ] ↓ [ Nettoyer + valider ] ↓ [ Ré-entraîner modèle ] ↓ [ Calculer métriques ] ↓ ┌────┴────┐ [ OK ] [ NON OK ] ↓ ↓ [ Promouvoir ] [ Alerter équipe ] en Production + garder ancien

Complexité Plus lourd à déployer Léger, cloud géré disponible
Observabilité Dashboard riche Dashboard moderne
Adoption Standard industrie Croissance rapide

Flashcards

Quelle est la différence entre une pipeline sklearn et un pipeline d'orchestration (Airflow/Prefect) ?

Voir la réponse
Réponse

Pipeline sklearn = chaîne de transformations sur des données (normaliser → encoder → prédire), tout s'exécute en mémoire en quelques secondes. Pipeline d'orchestration = séquence de tâches programmées (extraire → entraîner → valider → déployer), peut prendre des heures, gère les échecs, les retries et les dépendances entre tâches.

Pourquoi ne pas simplement utiliser un cron job plutôt qu'Airflow ou Prefect ?

Voir la réponse
Réponse

Un cron job lance un script — si ça échoue, personne ne le sait, pas de retry automatique, pas de visualisation des étapes, pas de gestion des dépendances. Airflow/Prefect ajoute : observabilité (voir quelle étape a échoué), retries (réessayer automatiquement), alertes, et dépendances (étape B ne démarre que si étape A a réussi).

65. Servir un modèle : FastAPI + Docker

Définition du serving : Servir un modèle signifie l'exposer comme un service — une application peut lui envoyer des données et recevoir une prédiction en retour. C'est la jonction entre le monde ML (un objet Python en mémoire) et le monde applicatif (une application web, mobile, ou un autre service qui a besoin de prédictions).

FastAPI : exposer un modèle en 20 lignes

FastAPI est une bibliothèque Python pour créer des services web rapides. Elle gère automatiquement la validation des données entrantes, la documentation interactive, et la sérialisation JSON.

# app.py
import mlflow
import pandas as pd
from fastapi import FastAPI
from pydantic import BaseModel  # validation automatique des données

app = FastAPI(title="Scoring Crédit BNP", version="1.0")

# Charger le modèle au démarrage (une seule fois)
model = mlflow.xgboost.load_model("models:/credit-scoring/Production")

# Définir le format des données attendues
class CreditRequest(BaseModel):
    age: int
    revenu_annuel: float
    anciennete_emploi: int
    montant_credit: float
    apport: float

class PredictionResponse(BaseModel):
    probabilite_defaut: float
    decision: str  # "accordé" ou "refusé"

@app.post("/predict", response_model=PredictionResponse)
def predict(request: CreditRequest):
    # Convertir en DataFrame (format attendu par le modèle)
    data = pd.DataFrame([request.dict()])

    # Prédiction
    proba = model.predict_proba(data)[0][1]  # probabilité de défaut
    decision = "refusé" if proba > 0.3 else "accordé"

    return PredictionResponse(
        probabilite_defaut=round(proba, 4),
        decision=decision
    )

@app.get("/health")
def health_check():
    return {"status": "ok", "model": "credit-scoring/Production"}

# Lancer le service (port 8000)
uvicorn app:app --host 0.0.0.0 --port 8000

# FastAPI génère automatiquement une documentation interactive sur :
# http://localhost:8000/docs

Docker : packager l'environnement

Définition : Docker est un outil qui encapsule une application et tout son environnement (Python, bibliothèques, configuration) dans un conteneur — une boîte isolée qui tourne de façon identique sur n'importe quelle machine. Sans Docker : "ça marche sur mon laptop" mais pas sur le serveur de prod (version de numpy différente, dépendance manquante). Avec Docker : l'environnement est figé et transportable.

# Dockerfile
FROM python:3.11-slim

WORKDIR /app

# Copier et installer les dépendances
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copier le code
COPY app.py .

# Exposer le port et démarrer
EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

# Construire l'image Docker
docker build -t credit-scoring-api:v1.0 .

# Lancer le conteneur
docker run -p 8000:8000 credit-scoring-api:v1.0

# Le service est maintenant accessible sur http://localhost:8000

Situation réelle : BNP Paribas, détection de fraude

Chaque transaction bancaire (carte, virement) doit être scorée en moins de 50ms — le client attend sa confirmation. Le modèle XGBoost tourne dans un conteneur Docker derrière un load balancer. À 50ms de latence maximale, un modèle de deep learning serait trop lent (200-500ms d'inférence CPU) ; XGBoost répond en 5-10ms.

Flashcards

Pourquoi utiliser Docker pour déployer un modèle ML ?

Voir la réponse
Réponse

Pour garantir que l'environnement est identique partout (dev, staging, prod). Sans Docker, une différence de version de numpy entre le laptop du data scientist et le serveur de prod peut suffire à faire planter le service. Docker fige l'environnement dans une image transportable.

Quand choisir batch inference plutôt que real-time ?

Voir la réponse
Réponse

Batch quand la prédiction n'est pas urgente (scorer les clients chaque nuit pour les campagnes marketing). Real-time quand la prédiction déclenche une action immédiate (valider une transaction, personnaliser une page web). Le coût d'infrastructure est très différent : le batch peut tourner sur une machine classique ; le real-time nécessite un service toujours actif avec SLA de latence.

76. Le monitoring en production

Définition : Le monitoring est la surveillance continue d'un système en production pour détecter les problèmes avant qu'ils deviennent critiques. Pour un modèle ML, il y a trois niveaux à surveiller simultanément : les métriques techniques (est-ce que le service répond ?), les métriques data (les données ont-elles changé ?), et les métriques métier (les prédictions produisent-elles toujours les bons résultats ?).

Ce qu'il faut surveiller

NIVEAU 1 — Technique (temps réel, alertes immédiates) ───────────────────────────────────────────────────── • Latence (P50, P95, P99) — combien de temps prend une prédiction • Taux d'erreur — % de requêtes qui échouent • Disponibilité — est-ce que le service répond ? • Volume de requêtes — pic ou chute anormale NIVEAU 2 — Data (hebdomadaire, alertes si dérive détectée) ────────────────────────────────────────────────────────── • Distribution des features d'entrée (PSI, test KS) • Valeurs manquantes inattendues • Nouvelles catégories non vues à l'entraînement NIVEAU 3 — Métier (mensuel, délai incompressible) ────────────────────────────────────────────────── • Performance réelle sur les décisions effectuées (nécessite d'attendre le résultat réel : défaut, churn, clic...) • Taux de conversion des recommandations • Coût des faux positifs / faux négatifs

Shadow mode : tester un nouveau modèle sans risque

Définition : Le shadow mode (ou shadow deployment) consiste à faire tourner un nouveau modèle en parallèle du modèle en production, sans que ses prédictions soient utilisées. Le nouveau modèle "observe" les requêtes réelles, ses prédictions sont loguées, mais c'est toujours l'ancien modèle qui agit.

Requête réelle ↓ ┌─────────────────┐ │ Modèle en PROD │ → Prédiction utilisée → Action réelle └─────────────────┘ ↓ (copie) ┌─────────────────┐ │ Nouveau modèle │ → Prédiction loguée → Jamais utilisée │ (shadow) │ └─────────────────┘

Après 1-2 semaines, on compare les distributions de prédictions des deux modèles. Si le nouveau est cohérent et meilleur → on le bascule en production. Si ses prédictions divergent fortement → on investigue avant de déployer.

Situation réelle : modèle de recommandation e-commerce

Un site e-commerce déploie un nouveau modèle de recommandation. Sans monitoring : les ventes issues des recommandations chutent de 15% sur 3 semaines avant que quelqu'un remarque que les recommandations sont devenues absurdes (le modèle recommande des produits en rupture de stock). Avec un monitoring métier hebdomadaire sur le taux de clic des recommandations, l'alerte aurait été déclenchée en 4 jours.

:::warning Le piège du monitoring technique seul : un service peut être techniquement parfait (latence 12ms, zéro erreur, disponibilité 100%) tout en produisant des prédictions complètement fausses. Les métriques techniques ne mesurent pas la qualité des prédictions — elles mesurent que le service répond. Le monitoring métier est obligatoire, même s'il est plus difficile à mettre en place. :::

Qu'est-ce que le shadow mode et pourquoi l'utiliser ?

Voir la réponse
Réponse

Faire tourner un nouveau modèle en parallèle du modèle en prod, sans que ses prédictions soient utilisées. But : valider le comportement du nouveau modèle sur des données réelles avant de le déployer. Permet de détecter des problèmes (prédictions aberrantes, latence trop élevée) sans risque pour la production.

Pourquoi les métriques techniques (latence, disponibilité) ne suffisent-elles pas pour monitorer un modèle ML ?

Voir la réponse
Réponse

Parce qu'elles mesurent que le service répond, pas que les prédictions sont bonnes. Un modèle peut répondre en 10ms avec zéro erreur tout en produisant des recommandations absurdes ou un scoring complètement dérivé. Le monitoring métier (taux de conversion, performance sur les décisions réelles) est obligatoire.

Quels sont les 3 niveaux de monitoring à mettre en place pour un modèle en production ?

Voir la réponse
Réponse

1. Technique (temps réel) : latence, taux d'erreur, disponibilité. 2. Data (hebdomadaire) : drift des features d'entrée via Evidently/PSI. 3. Métier (mensuel, avec délai) : performance réelle sur les décisions prises — nécessite d'attendre les résultats observables.

1. MLOps = traçabilité + reproductibilité + automatisation
   80% des modèles ne passent pas en prod sans ces 3 éléments

3. Data drift ≠ Model drift
   Data drift : les ENTRÉES changent
   Model drift : la RELATION entrées/cible change
   Détecter : Evidently + PSI (seuil > 0.2 → action)

4. Orchestration = DAG de tâches (Airflow ou Prefect)
   PAS une pipeline sklearn — un chef d'orchestre qui planifie
   et gère les échecs automatiquement

5. Serving = FastAPI (exposer) + Docker (packager l'environnement)
   Batch = prédictions en masse, pas urgent
   Real-time = <100ms, déclenche une action immédiate

6. Monitoring = 3 niveaux obligatoires
   Technique (temps réel) + Data (hebdo) + Métier (mensuel)
   Le shadow mode permet de tester sans risquer la prod
