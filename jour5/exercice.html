import mlflow
import xgboost as xgb
import pandas as pd
from sklearn.model_selection import train_test_split
from sklearn.metrics import accuracy_score, f1_score

X = pd.DataFrame({'age': [25,35,45,55,28,62], 'revenu': [30000,50000,70000,90000,35000,45000]})
y = [0, 0, 1, 1, 0, 1]
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.3, random_state=42)

# TODO 1
mlflow.set_experiment("credit-scoring-experiment")

# TODO 2
with mlflow.start_run(run_name="xgb-lr0.1-n100"):

    params = {"n_estimators": 100, "learning_rate": 0.1, "max_depth": 4}

    # TODO 3
    mlflow.log_params(params)

    model = xgb.XGBClassifier(**params)
    model.fit(X_train, y_train)
    y_pred = model.predict(X_test)

    acc = accuracy_score(y_test, y_pred)
    f1 = f1_score(y_test, y_pred)

    # TODO 4
    mlflow.log_metric("accuracy", acc)
    mlflow.log_metric("f1_score", f1)

    # TODO 5
    mlflow.xgboost.log_model(model, "model")

    print(f"Run terminé - Accuracy: {acc:.3f}, F1: {f1:.3f}")

# TODO 6
run_id = mlflow.last_active_run().info.run_id
mlflow.register_model(
    model_uri=f"runs:/{run_id}/model",
    name="credit-model"
)

# TODO 7 — Note : en pratique, passer d'abord en Production via le Registry
# mlflow.MlflowClient().transition_model_version_stage("credit-model", "1", "Production")
loaded_model = mlflow.xgboost.load_model("models:/credit-model/Production")
predictions = loaded_model.predict(X_test)
print(f"Prédictions : {predictions}")

Points clés : - log_params prend un dict, log_metric prend clé+valeur - Le model_uri dans register_model suit le format runs:/{run_id}/artifact_path - Pour charger depuis le Registry : models:/nom-modele/stage - Il faut passer le modèle en "Production" avant de pouvoir le charger avec ce stage

Exercice 3 — Diagnostiquer un drift (15 min)

Un modèle de scoring de crédit a été entraîné en janvier 2023. Voici les distributions comparées de deux features clés entre les données d'entraînement et les données de production en janvier 2025 :

Feature 1 : type_emploi

Valeur Train 2023 Prod 2025
CDI 68% 52%
CDD 18% 21%
Indépendant 8% 19%
Sans emploi 6% 8%

Feature 2 : montant_credit_demande (€)

Statistique Train 2023 Prod 2025
Médiane 12 000 18 500
P90 45 000 72 000
Max 150 000 380 000

Questions : 1. Quelle feature a le plus drifté ? Pourquoi est-ce problématique ? 2. De quel type de drift s'agit-il (data drift ou model drift) ? 3. Quelle action concrète prendre ?

Réponse

1. Feature la plus driftée : type_emploi

Les indépendants sont passés de 8% à 19% — soit +138%. Le modèle a été entraîné avec très peu d'exemples d'indépendants, donc ses prédictions sur cette catégorie sont peu fiables (petit échantillon d'apprentissage → haute variance). montant_credit_demande a aussi drifté (médiane +54%) mais reste dans un continuum numérique où le modèle peut extrapoler ; la catégorie Indépendant est plus risquée car c'est une catégorie sous-représentée à l'entraînement.

2. Type de drift : data drift

Les données d'entrée ont changé (explosion des profils d'indépendants, probablement liée au développement du freelancing et du statut d'auto-entrepreneur post-COVID). Ce n'est pas encore confirmé comme model drift — la relation entre ces features et le risque de défaut a peut-être aussi changé, mais on ne peut pas le confirmer sans mesurer les performances réelles.

3. Actions à prendre : - Court terme : augmenter la surveillance sur les dossiers Indépendant (revue humaine systématique) - Moyen terme : collecter des labels récents (résultats sur les dossiers accordés en 2024) et ré-entraîner avec un jeu de données plus représentatif de la réalité actuelle - Monitoring : calculer le PSI sur type_emploi (probablement > 0.25 → action immédiate requise) et ajouter une alerte automatique si PSI dépasse 0.2

Question ouverte : Une entreprise veut déployer un modèle de churn client. Elle a MLflow mais pas d'orchestration, pas de monitoring. Quel est le risque le plus critique à adresser en premier, et pourquoi ?

Réponse suggérée

Le monitoring est le risque le plus critique, devant l'orchestration.

Sans monitoring, le modèle peut se dégrader silencieusement pendant des mois sans que personne le sache. Les campagnes de rétention client ciblées par le modèle deviendront progressivement inefficaces, avec un coût croissant. Le business ne saura pas que c'est le modèle le problème — il pensera que les clients sont "moins réceptifs".

Sans orchestration, le ré-entraînement est manuel mais au moins contrôlé. C'est un problème d'efficacité, pas de fiabilité.

Priorité pratique : 1. Monitoring métier d'abord (mesurer le taux de churn réel des clients flaggés vs non-flaggés) 2. Monitoring data (détecter le drift avant que les performances chutent) 3. Orchestration (automatiser ce qui est actuellement fait manuellement)
