m = Prophet(

Exercice 3 — Cas piège : data leakage temporel (25 min)

Le code suivant contient une erreur grave de validation. Identifie le problème et corrige-le.

# CODE AVEC ERREUR — à corriger
import pandas as pd
import numpy as np
from prophet import Prophet
from sklearn.model_selection import train_test_split

# Série temporelle : consommation d'eau mensuelle sur 5 ans (60 mois)
dates = pd.date_range("2019-01", periods=60, freq="ME")
np.random.seed(1)
consommation = 200 + 30 * np.sin(np.linspace(0, 10*np.pi, 60)) + np.random.normal(0, 5, 60)

df = pd.DataFrame({"ds": dates, "y": consommation})

# ─── Split aléatoire (problématique) ───────────────────────────────────────
train, test = train_test_split(df, test_size=0.2, random_state=42)
# train : 48 lignes aléatoires parmi 60
# test  : 12 lignes aléatoires parmi 60

# Entraînement
m = Prophet(yearly_seasonality=True)
m.fit(train)

# Prévision sur les dates de test
futur = pd.DataFrame({"ds": test["ds"]})
forecast = m.predict(futur)

# Évaluation
mae = np.mean(np.abs(test["y"].values - forecast["yhat"].values))
print(f"MAE : {mae:.2f}")  # Affiche 4.2 — semble excellent !

Correction et explication

1. Le problème précis

train_test_split avec shuffle=True (comportement par défaut) mélange aléatoirement les 60 lignes. Le jeu de test contient donc des mois de 2019, 2020, 2021, 2022, 2023 mélangés — et le jeu d'entraînement aussi. Prophet est entraîné sur des données de tout le cycle temporel, y compris des données plus récentes que certaines données de test.

Par exemple, le mois test 2023-06 (le plus récent) est prédit par un modèle qui a vu 2023-07, 2023-08, etc. pendant l'entraînement. C'est du data leakage temporel : le modèle a accès à des informations du futur.

2. Pourquoi la MAE de 4.2 est trompeuse

La série est une sinusoïde propre avec peu de bruit. Un modèle qui a vu des points de toute la période apprend exactement la forme du cycle. Les points de test, disséminés dans le cycle, sont quasi-interpolés entre des points d'entraînement voisins dans le temps. La vraie question est : si le modèle n'avait vu que les 48 premiers mois, arriverait-il à prédire les 12 derniers aussi bien ?

3. Correction

# Split temporel correct
# On garde les 48 premiers mois pour l'entraînement,
# et les 12 derniers (1 an complet) pour le test

train = df.iloc[:48]   # 2019-01 à 2022-12
test  = df.iloc[48:]   # 2023-01 à 2023-12

m = Prophet(yearly_seasonality=True)
m.fit(train)

futur = m.make_future_dataframe(periods=12, freq="ME")
forecast = m.predict(futur).tail(12)

mae = np.mean(np.abs(test["y"].values - forecast["yhat"].values))
print(f"MAE corrigée : {mae:.2f}")

4. MAE après correction

Elle sera probablement similaire ici (4-7) car la série est propre et régulière. Mais sur des séries réelles avec tendance non-stationnaire ou événements imprévus, la MAE après correction temporelle est systématiquement plus élevée — parfois 3 à 5 fois — que la MAE du split aléatoire. C'est cette MAE corrigée qui reflète la performance réelle en production.

Règle à retenir : sur une série temporelle, ne jamais utiliser train_test_split sans shuffle=False et sans ordonner les données chronologiquement. Préférer un découpage manuel ou TimeSeriesSplit.

Q1 — Un collègue te montre un modèle Prophet avec un MAPE de 1.8% sur le backtesting. Il te dit "c'est excellent, on peut le déployer". Quelle question cruciale tu lui poses ?

Réponse

La question cruciale est : "Ce MAPE est-il mesuré in-sample (sur les données d'entraînement) ou par walk-forward validation sur un jeu de test strictement postérieur ?" Un MAPE in-sample peut résulter de l'overfitting — le modèle a mémorisé les données plutôt que d'en apprendre les patterns. Il faut également demander sur quel horizon de prévision ce MAPE est calculé.

Q2 — Un client te dit "j'ai des données de ventes de 50 magasins sur 3 ans, chaque vente est influencée par la météo locale et par les promotions nationales. Je veux prévoir à 4 semaines." Lequel choisirais-tu parmi statsmodels, Prophet ou darts, et pourquoi ?

Réponse

Prophet est le bon choix ici. Il gère nativement plusieurs saisonnalités (hebdomadaire + annuelle), les régresseurs externes (météo via add_regressor, promotions via un indicateur binaire), et peut être entraîné indépendamment pour chacun des 50 magasins sans configuration ARIMA manuelle. darts serait justifié si on voulait capturer des interactions entre magasins (modèle multivarié) ou si on avait besoin d'intervalles de confiance calibrés pour des décisions d'approvisionnement critiques. statsmodels serait trop manuel à configurer pour 50 séries parallèles.

Q3 — Explique en une phrase ce qu'est l'autocorrélation, comme si tu devais l'expliquer à un responsable métier non-technique.

Réponse

"L'autocorrélation mesure à quel point les ventes d'un lundi ressemblent à celles du lundi précédent — si la réponse est 'beaucoup', on dit que la série a une autocorrélation forte à 7 jours, et les algorithmes peuvent exploiter ce pattern pour prévoir."

Q4 — Quelle est la différence entre une anomalie et une saisonnalité dans une série temporelle ?

Réponse

Une saisonnalité est un cycle régulier, prévisible et récurrent : les ventes remontent chaque décembre, les pics de trafic reviennent chaque lundi matin. Une anomalie est un événement ponctuel non récurrent : une panne système, une grève, une vague de chaleur exceptionnelle. La saisonnalité enrichit le modèle (elle doit être capturée). L'anomalie doit être identifiée et traitée avant l'entraînement (suppression ou annotation explicite) pour ne pas biaiser les paramètres du modèle.

Q5 — Quel est le risque de choisir model="additive" dans seasonal_decompose sur une série de ventes d'e-commerce qui a doublé en 2 ans ?

Réponse

Si les ventes ont doublé, l'amplitude des pics saisonniers (Noël, soldes) a probablement aussi doublé — c'est le signe d'une décomposition multiplicative. Forcer le modèle additif revient à supposer que les pics gardent une amplitude constante de, disons, +500 ventes, alors qu'ils sont passés de +500 à +1000. Les résidus contiendront une structure systématique (sous-estimation des récents pics, sur-estimation des anciens), ce qui signifie que le modèle rate une information exploitable et que ses prévisions de pics seront biaisées.
