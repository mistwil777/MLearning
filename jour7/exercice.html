Exercice unique — Malakoff Humanis : Réduire le taux de résiliation

Le brief client

Malakoff Humanis est une mutuelle de santé. Leur taux de résiliation annuel est de 8,2%, en hausse depuis 3 ans. Chaque point de résiliation représente environ 15M€ de primes perdues. La direction veut "utiliser l'IA pour retenir les clients qui vont partir".

Ce que vous avez : - 2 millions de contrats actifs avec 5 ans d'historique complet - Données de sinistres : fréquence de remboursement, type (médecin, hospitalisation, dentaire, optique), montants - Données d'appels au service client : nombre, durée, motif (réclamation / information / résiliation directe), résolution - Données de changements de situation : changement d'employeur, départ en retraite, changement de situation familiale, déménagement - Données économiques : variation du revenu estimé (source DSN — Déclaration Sociale Nominative) - Données de navigation sur l'espace client en ligne

L'équipe : - 3 data scientists (dont vous, junior) - 1 data engineer - Pas de MLOps dédié

Contrainte réglementaire : toute décision commerciale basée sur ce modèle (appel de rétention, offre personnalisée) doit être explicable. La CNIL et le code des assurances imposent que le client puisse comprendre pourquoi il a été contacté.

Ce que vous devez produire

Répondez aux 5 parties ci-dessous. Chaque partie a une réponse commentée que vous pouvez déplier après votre tentative.

Partie 1 — Reformuler le problème métier

Question : "Réduire le taux de résiliation via l'IA" est une intention, pas un problème ML. Reformulez-le en problème ML valide (rappel : variable à prédire, contexte d'utilisation, critère de succès mesurable).

Notez votre reformulation ici avant de déplier la correction.

Réponse commentée — Partie 1

Reformulation correcte :

Prédire, pour chaque contrat actif, la probabilité de résiliation dans les 90 prochains jours, afin de déclencher une action de rétention personnalisée (appel, offre de rachat, amélioration de garanties) auprès des contrats à risque élevé, avant que la résiliation soit initiée.

Pourquoi 90 jours et pas 30 ou 12 mois ?

  • 30 jours : trop court pour déclencher une intervention (traitement de l'alerte, formation des conseillers, temps d'appel) — on serait en réaction.
  • 12 mois : trop long — la plupart des événements déclencheurs n'ont pas encore eu lieu, le modèle prédit du vent.
  • 90 jours : fenêtre standard dans le secteur assurance. Assez long pour agir, assez court pour que les signaux soient pertinents.

Variable cible : résiliation dans les 90 prochains jours (binaire : 0 = reste, 1 = résilie)

Contexte d'utilisation : scoring hebdomadaire de l'ensemble du portefeuille, résultat envoyé aux équipes de rétention sous forme de liste priorisée

Critère de succès métier : réduire le taux de résiliation de 8,2% à 6,5% sur les contrats "traités" par l'équipe de rétention (attention : pas du taux global — les 3 data scientists ne peuvent pas traiter 2M de contrats)

Critère de succès technique : AUC > 0.80 en validation, recall sur les 10% de contrats les plus à risque > 70%

Partie 2 — Audit des données

Question : Parmi les sources de données disponibles, identifiez : 1. Deux sources qui seront probablement de haute valeur prédictive 2. Deux risques concrets sur la qualité ou la légalité des données 3. Un signal qui manque et qui serait précieux

Réponse commentée — Partie 2

Sources à haute valeur prédictive :

1. Données d'appels au service client C'est le signal le plus fort. Un client qui appelle pour une réclamation non résolue, puis rappelle deux semaines plus tard, puis appelle directement le service résiliation : ce pattern de 3 appels en 30 jours prédit la résiliation avec une précision très élevée dans tous les datasets du secteur assurance. Le motif "résiliation directe" est presque un label en lui-même.

2. Données de changements de situation Le départ en retraite est un événement massif : le contrat collectif d'entreprise ne suit pas. La personne doit souscrire un contrat individuel — souvent chez un concurrent plus agressif commercialement. Même logique pour un déménagement en zone de faible couverture réseau médecin. Ces événements ont un fort pouvoir prédictif car ils créent une fenêtre de vulnérabilité contractuelle.

Deux risques identifiés :

Risque 1 — Qualité des motifs d'appel Les motifs sont saisis manuellement par les conseillers. En pratique, "réclamation" et "information" sont souvent confondus, et certains conseillers utilisent "autre" pour tout ce qui prend du temps à catégoriser. Une feature construite sur des motifs mal encodés introduit du bruit. Il faut auditer la distribution de chaque motif par conseiller.

Risque 2 — Biais de sélection dans les labels On n'observe la résiliation que pour les clients qui l'ont initiée. Mais des clients insatisfaits qui n'ont pas encore résilié sont dans les négatifs (y=0). Si on entraîne le modèle sans distinguer "satisfait" de "insatisfait qui n'a pas encore agi", on sous-estime le risque de ce deuxième groupe. Le modèle sera moins performant sur les silencieux.

Signal manquant précieux : Les données de comparaison en ligne — est-ce que le client a consulté des comparateurs d'assurance (ASSUREo, LeLynx) récemment ? Ce signal n'est pas disponible directement, mais il pourrait être approximé par des partenariats de données ou des enquêtes NPS envoyées au bon moment. Son absence est un vrai angle mort du modèle.

Partie 3 — Architecture technique

Question : Proposez une architecture complète. Justifiez explicitement : - Pourquoi vous utilisez (ou n'utilisez pas) un LLM - Pourquoi vous choisissez l'algorithme de scoring central - Comment vous gérez la contrainte d'explicabilité réglementaire

Réponse commentée — Partie 3

Architecture proposée :

Pourquoi XGBoost et pas un LLM ou un réseau de neurones ?

La contrainte réglementaire est déterminante. La CNIL impose l'explicabilité des décisions automatisées impactant les personnes (Article 22 du RGPD). XGBoost avec SHAP permet de produire pour chaque contrat les 3 facteurs qui contribuent le plus au score — texte compréhensible par un conseiller et présentable au client.

Un réseau de neurones profond serait plus performant en AUC (+2-3 points estimés) mais ininterprétable nativement. Un LLM ajouterait de la latence, du coût, et ne s'entraîne pas sur des données tabulaires structurées.

Pourquoi pas de règles seules ? Les règles simples ("si > 3 appels en 30 jours → alerte") ont un rappel faible. Elles ne capturent pas les patterns multivariés : un client avec 2 appels ET un déménagement ET une baisse de remboursement est plus à risque qu'un client avec 4 appels et aucun autre signal.

Comment SHAP produit les explications :

import shap
import xgboost as xgb

modele = xgb.XGBClassifier(...)
modele.fit(X_train, y_train)

explainer = shap.TreeExplainer(modele)
shap_values = explainer(X_test)

def expliquer_contrat(contrat_id, shap_values, feature_names):
    """Retourne les 3 facteurs principaux pour un contrat."""
    sv = shap_values[contrat_id].values
    top3_idx = abs(sv).argsort()[-3:][::-1]

    explications = []
    for idx in top3_idx:
        feature = feature_names[idx]
        contribution = sv[idx]
        direction = "augmente" if contribution > 0 else "réduit"
        explications.append(
            f"'{feature}' {direction} le risque de résiliation"
        )
    return explications

# Exemple de sortie :
# ['nb_appels_30j=4 augmente le risque de résiliation',
#  'motif_dernier_appel=reclamation augmente le risque',
#  'remboursements_dentaire_12m=0 augmente le risque']

Ce texte est directement utilisable par le conseiller lors de l'appel de rétention.

Partie 4 — Métriques de succès

Question : Définissez les métriques que vous proposez à la direction, en distinguant métriques techniques et métriques métier. Justifiez pourquoi vous ne proposez pas l'exactitude (accuracy) comme métrique principale.

Réponse commentée — Partie 4

Pourquoi l'exactitude est trompeuse ici :

Le taux de résiliation est de 8,2% — donc 91,8% des contrats ne résilient pas. Un modèle stupide qui prédit "jamais de résiliation" atteint une accuracy de 91,8%. C'est excellent sur papier et complètement inutile en pratique.

Métriques techniques :

Métrique Valeur cible Pourquoi
AUC-ROC > 0.80 Mesure la capacité du modèle à distinguer les résilient des non-résiliant, indépendamment du seuil de décision
Recall sur les 10% de contrats les plus à risque > 65% On veut être sûr de capturer la majorité des "vrais" résiliants dans le top du classement — c'est ce que l'équipe de rétention traitera
Precision@top10% > 35% Au moins 35% des contrats alertés doivent être de "vrais" résiliants — pour que l'équipe ne passe pas 65% de son temps sur des faux positifs

Pourquoi recall et precision@k plutôt que F1 global ?

Parce que l'équipe de rétention ne peut pas appeler 2M de contrats. Elle appelle les 5 à 10 000 contrats avec le score le plus élevé. Ce qui compte n'est pas la performance globale du modèle mais sa performance dans ce sous-ensemble. C'est ce que capture le recall et la precision sur les top-k contrats.

Métriques métier :

Métrique Baseline Cible Mesure
Taux de résiliation sur contrats "traités" 8,2% 5,5% Comparer le taux de résiliation à 90j sur les contrats contactés vs non-contactés (A/B test interne)
ROI de la campagne de rétention — > 3× Primes conservées / coût des actions de rétention (appels, offres)
Taux de conversion des offres de rétention — > 20% Parmi les contrats contactés, % qui renouvellent avec une offre améliorée

Indicateur de vigilance :

Taux de faux positifs traités — si les conseillers passent trop de temps sur des faux positifs (clients qui ne résiliaient pas), ils perdront confiance dans le modèle. Ce taux doit être suivi mensuellement et partagé avec l'équipe.

Partie 5 — Plan de mise en production

Question : Proposez un plan de mise en production réaliste pour une équipe de 3 data scientists + 1 data engineer, sans MLOps dédié. Incluez les outils, la cadence, et ce que vous monitorez.

Réponse commentée — Partie 5

Contrainte principale : pas de MLOps dédié = pas de Kubernetes, pas d'infrastructure complexe à maintenir. Chaque heure passée à maintenir l'infra est une heure de moins sur le modèle.

Phase 1 — POC (semaines 1-4)

Objectif : prouver que le modèle a un AUC > 0.80 sur données historiques. Pas de déploiement, pas de pipeline automatisé. Un notebook reproductible suffit.

Livrable : rapport avec AUC, matrice de confusion, top features SHAP, estimation du ROI. Ce rapport convainc ou non la direction d'investir en phase 2.

Phase 2 — Déploiement minimal (semaines 5-10)

{ } Code
Chaque lundi à 3h du matin :
  1. Airflow déclenche un DAG
  2. Le DAG extrait les données de la semaine (SQL → CSV)
  3. Le DAG applique le modèle XGBoost pré-entraîné
  4. Le DAG génère une liste CSV : contrat_id, score, top3_raisons
  5. Le CSV atterrit dans un Google Sheet partagé avec l'équipe rétention
  6. L'équipe rétention appelle les N contrats les plus à risque

Pourquoi Google Sheet ? Parce que l'équipe rétention n'a pas accès à un outil de CRM avancé aujourd'hui. La simplicité garantit l'adoption.

Phase 3 — Ré-entraînement (mois 3+)

Le modèle est ré-entraîné mensuellement sur les données cumulées. Un modèle challenger est entraîné en parallèle et comparé au champion en validation avant déploiement.

MLflow pour tracker chaque expérience (paramètres, métriques, version du dataset).

Outils retenus :

Besoin Outil Justification
Orchestration Apache Airflow Standard, cloud-native, gratuit en self-hosted
Tracking expériences MLflow Open source, intégration native XGBoost
Stockage modèles MLflow Model Registry Versionning, pas besoin d'infra dédiée
Explicabilité SHAP Natif XGBoost, rapide
Monitoring dérive Evidently AI Open source, alertes email simples

Ce que vous monitorez :

Signal Outil Seuil d'alerte Action
Distribution du score moyen Evidently ±15% vs mois précédent Retraining immédiat
AUC sur les 4 dernières semaines MLflow < 0.75 Investigation + retraining
Taux de résiliation réel vs prédit Dashboard Airflow Écart > 2 points Révision des features
Qualité des données en entrée Airflow DAG Null > 20% sur une feature clé Alerte data engineer

Estimation du coût total de l'infrastructure :

  • Serveur Airflow + MLflow (cloud, instance moyenne) : ~150€/mois
  • Stockage données : ~50€/mois
  • Total infra : ~200€/mois
  • Comparaison : retenir 1% de plus du portefeuille = ~15M€ de primes conservées
