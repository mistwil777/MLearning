21. Méthode : du brief client à l'architecture technique

Définition. Un projet ML ne commence jamais par le code. Il commence par une demande — souvent vague, parfois contradictoire — qu'il faut transformer en problème résolvable par une machine. La méthode ci-dessous est un processus en 5 étapes qui structure ce travail de traduction.

:::situation 📍 Situation réelle — La demande vague

Un directeur digital de grande surface vous convoque. Il dit : "On veut faire de l'IA pour améliorer l'expérience client." Silence. Il attend. Il ne sait pas ce qu'il veut exactement, mais il a lu un article sur Amazon. Votre réponse à cet instant définit la qualité de tout le projet qui suivra. :::

Étape 1 — Reformuler le problème métier en problème ML

La plupart des demandes métier sont des symptômes. "Améliorer l'expérience client" est un objectif business, pas un problème ML. Pour le transformer, il faut poser les bonnes questions :

  • Quel comportement client vous coûte de l'argent en ce moment ?
  • Pouvez-vous me donner un exemple concret de "mauvaise expérience" ?
  • Si vous deviez mesurer le succès dans 6 mois, que regarderiez-vous ?

Ces questions font émerger un problème précis : "30% des clients abandonnent leur panier avant de payer, et on ne sait pas pourquoi." C'est un problème ML : prédire l'abandon de panier pour déclencher une intervention en temps réel.

La règle de reformulation. Un problème ML valide contient : une variable à prédire, un contexte d'utilisation, et un critère de succès mesurable. "Améliorer l'expérience" ne contient aucun des trois.

Étape 2 — Auditer les données disponibles

Définition. Un audit de données répond à quatre questions : Est-ce que les données existent ? Sont-elles de bonne qualité ? Couvrent-elles le phénomène qu'on veut prédire ? Peut-on les utiliser légalement ?

Dimension Questions à poser Signal d'alarme
Existence Les données sont-elles collectées ? Depuis quand ? Données < 6 mois pour un phénomène annuel
Qualité Taux de valeurs manquantes ? Erreurs de saisie ? > 20% de valeurs manquantes sur des features clés
Labels La variable cible est-elle disponible ? Fiable ? Labels créés manuellement par des humains fatigués
Légalité RGPD, consentement, données sensibles ? Données de santé sans DPO consulté
Volume Assez d'exemples du phénomène rare ? 0,1% d'événements positifs sur 10k lignes
Fraîcheur Les données de 2019 reflètent-elles 2024 ? Data pre-Covid pour prédire comportements post-Covid

:::warning ⚡ Piège — Le biais de confirmation dans l'audit

Il est tentant de déclarer les données "suffisantes" pour ne pas décevoir le client. C'est la principale cause d'échec de projets ML. Un projet lancé sur des données insuffisantes échouera dans 6 mois avec 5× plus de coût qu'un refus honnête dès le départ. :::

Étape 3 — Choisir l'architecture

Le choix de l'architecture est la décision la plus structurante. Elle dépend de l'audit des données, de la contrainte de latence, du besoin d'explicabilité et du budget.

Arbre de décision simplifié :

def choisir_architecture(problem):
    # Règles d'abord — souvent sous-estimées
    if problem.regles_metier_claires and problem.volume_faible:
        return "Règles if/else + validation humaine"

    # ML classique — le cheval de bataille
    if problem.donnees_tabulaires and problem.volume < 10_000_000:
        if problem.interpretabilite_requise:
            return "Logistic Regression ou Decision Tree"
        return "XGBoost (Gradient Boosting)"

    # Séries temporelles
    if problem.type == "prévision_temporelle":
        if problem.patterns_saisonniers:
            return "Prophet + XGBoost pour anomalies"
        return "LSTM ou Transformer temporel"

    # Deep Learning — seulement si données non-structurées
    if problem.donnees_images or problem.donnees_texte:
        if problem.budget_inference_eleve:
            return "Fine-tuning LLM"
        return "BERT/CamemBERT classifié"

    # Hybride — le cas le plus fréquent en entreprise
    return "Pipeline hybride : ML classique + règles métier"

:::rule ✅ Règle d'or — La complexité doit être justifiée

Un LLM coûte entre 100× et 1000× plus cher à l'inférence qu'un XGBoost. Avant de le choisir, il faut démontrer qu'un modèle plus simple ne suffit pas. Ce n'est pas un choix par défaut. :::

Étape 4 — Définir les métriques de succès

Définition. Une métrique est une mesure quantitative qui dit si un modèle remplit son objectif. Il en faut deux types : les métriques techniques (précision, rappel, AUC…) et les métriques métier (chiffre d'affaires supplémentaire, taux de réduction des incidents, temps économisé).

La confusion entre les deux est une source fréquente de conflits. Un modèle peut avoir une excellente AUC et un impact métier nul, parce que les décisions prises sur sa base n'étaient pas les bonnes.

Contexte Métrique technique Métrique métier
Détection de fraude Recall sur fraudes (minimiser faux négatifs) Montant frauduleux évité / mois
Recommandation NDCG@10 (classement des suggestions) CTR, panier moyen, conversions
Prévision de stock MAE (écart absolu moyen) Coût de rupture + coût de surstock
Classification texte F1-macro Temps économisé par les agents

Étape 5 — Planifier le cycle de vie

Définition. Un modèle ML n'est pas un livrable statique. C'est un système vivant qui se dégrade avec le temps parce que le monde change. Le cycle de vie couvre : entraînement initial, déploiement, monitoring, et ré-entraînement.

Entraînement ──► Validation ──► Déploiement ──► Monitoring ▲ │ └─────────────── Ré-entraînement ◄──────────────┘ (dérive détectée)

La cadence de ré-entraînement dépend de la vitesse à laquelle le phénomène évolue : quotidien pour la détection de fraude, hebdomadaire pour la prévision de stock, mensuel pour la recommandation.

Flashcards

Quelle est la différence entre une métrique technique et une métrique métier ? Donnez un exemple.

Voir la réponse
Réponse

Une métrique technique mesure la performance du modèle (ex : AUC = 0.87 pour détecter la fraude). Une métrique métier mesure l'impact réel (ex : 450 000 € de fraude évités par mois). Un modèle peut avoir une excellente AUC et un impact métier nul si les décisions prises sur sa base sont mauvaises.

Pourquoi un LLM n'est pas le bon choix par défaut pour un problème de classification ?

Voir la réponse
Réponse

Un LLM coûte 100× à 1000× plus cher à l'inférence qu'un XGBoost. Pour un problème de classification sur données tabulaires, XGBoost est plus rapide, plus interprétable, et plus précis. Le LLM ne se justifie que si le problème nécessite de la compréhension de texte libre ou de la génération.

Qu'est-ce qu'un audit de données et quand faut-il refuser de démarrer un projet ?

Voir la réponse
Réponse

Un audit de données vérifie l'existence, la qualité, les labels, la légalité et le volume. Il faut refuser de démarrer si les labels sont absents ou peu fiables, si les données ne couvrent pas le phénomène prédit, ou si des contraintes RGPD n'ont pas été traitées par le DPO.

Que contient un problème ML "valide" selon la règle de reformulation ?

Voir la réponse
Réponse

Un problème ML valide contient trois éléments : une variable à prédire (ex : abandon de panier), un contexte d'utilisation (ex : en temps réel pendant la navigation), et un critère de succès mesurable (ex : réduire le taux d'abandon de 30% à 22%).

32. Cas complet 1 — Système de recommandation e-commerce (Fnac)

Le brief

La Fnac veut "personnaliser l'expérience d'achat". Après reformulation : augmenter le taux de conversion sur la page produit en proposant des articles pertinents à chaque visiteur.

Données disponibles : historique d'achats (3 ans, 8M transactions), navigation (clics, temps passé), catalogue produits (500k références), données démographiques (âge, région, programme fidélité). Labels : achat = positif, navigation sans achat = négatif (proxy imparfait).

Les 5 étapes appliquées

Étape 1 — Reformulation. Prédire pour chaque utilisateur et chaque session les 5 produits les plus susceptibles d'être achetés, parmi ceux qu'il n'a pas encore achetés.

Étape 2 — Audit. Données riches, 3 ans d'historique. Problème identifié : cold start (30% des visiteurs sont nouveaux, sans historique). Les labels "navigation sans achat" sont du bruit : on peut naviguer longtemps sans vouloir acheter.

Étape 3 — Architecture choisie.

┌─────────────────────────────────┐ │ PIPELINE RECOMMANDATION FNAC │ └──────────────┬──────────────────┘ │ ┌────────────────────┼────────────────────┐ ▼ ▼ ▼ ┌─────────────────┐ ┌──────────────────┐ ┌─────────────┐ │ Embedding │ │ Filtrage │ │ Règles │ │ produits │ │ collaboratif │ │ métier │ │ (Word2Vec-like) │ │ (ALS / Matrix │ │ (nouveautés,│ │ sur co-achats │ │ Factorization) │ │ stock dispo)│ └────────┬────────┘ └────────┬─────────┘ └──────┬──────┘ └────────────────────┼───────────────────┘ ▼ ┌────────────────────────┐ │ Ranking final : XGBoost │ │ (features : score reco, │ │ marge, fraîcheur, CTR │ │ historique) │ └───────────┬─────────────┘ ▼ ┌──────────────────┐ │ A/B Test │ │ (10% trafic) │ └──────────────────┘

Pourquoi pas un LLM ici ? Un LLM génératif pourrait "décrire" des produits recommandés, mais il ne peut pas apprendre les corrélations entre 8M transactions. Le filtrage collaboratif exploite exactement ce que le LLM ne peut pas voir : "les gens qui achètent X achètent aussi Y".

Pourquoi XGBoost pour le ranking final ? Parce que le ranking intègre des features métier (marge produit, niveau de stock, partenariats commerciaux) que les algorithmes de recommandation pure n'optimisent pas. XGBoost les combine proprement et reste interprétable pour la direction commerciale.

Code skeleton — Embedding produits

# Embedding produits via co-achats (style Word2Vec)
from gensim.models import Word2Vec

# Chaque "phrase" = liste ordonnée des achats d'un panier
baskets = [
    ["livre_polar", "stylo_luxe", "carnet"],
    ["casque_audio", "cable_usb", "pochette"],
    ...
]

model = Word2Vec(
    sentences=baskets,
    vector_size=128,    # dimension de l'embedding
    window=5,           # contexte dans le panier
    min_count=10,       # ignorer les produits rares
    workers=4
)

# Produits similaires à un article donné
similaires = model.wv.most_similar("casque_audio", topn=10)
# [('casque_gaming', 0.92), ('micro_usb', 0.88), ...]

Code skeleton — Ranking XGBoost

import xgboost as xgb
import pandas as pd

# Features pour le ranking
features = [
    'score_collaboratif',    # issu du filtrage collaboratif
    'similarite_embedding',  # distance cosinus dans l'espace produit
    'marge_produit',         # feature métier
    'stock_disponible',      # règle binaire
    'ctr_historique',        # taux de clic passé sur ce produit
    'fraicheur_article',     # jours depuis la mise en ligne
]

X_train = df[features]
y_train = df['achete']  # 1 = acheté, 0 = non-acheté

model = xgb.XGBClassifier(
    n_estimators=300,
    max_depth=6,
    learning_rate=0.05,
    eval_metric='auc',
)
model.fit(X_train, y_train)

Métriques et monitoring

  • Métrique technique : NDCG@5 (qualité du classement des 5 premières suggestions)
  • Métrique métier : taux de conversion sur page produit (baseline : 2.3%, cible : 3.0%)
  • Monitoring : dérive de distribution des embeddings (les produits tendance changent vite)
  • Ré-entraînement : hebdomadaire pour les embeddings, quotidien pour le ranking XGBoost
Flashcards

Qu'est-ce que le "cold start" en recommandation et comment le résoudre ?

Voir la réponse
Réponse

Le cold start est le problème posé par les nouveaux utilisateurs sans historique : impossible de calculer un filtrage collaboratif. Solutions : recommander les tops ventes par catégorie, utiliser les données démographiques disponibles (âge, région), ou démarrer avec un questionnaire d'onboarding.

Pourquoi utilise-t-on des embeddings de produits plutôt qu'une simple similarité de catégorie ?

Voir la réponse
Réponse

Les catégories sont définies par des humains (ex : "Électronique"). Les embeddings apprennent les associations réelles dans les données (ex : les acheteurs de casques audio achètent aussi des câbles USB et des coques de téléphone — hors même catégorie). Ils capturent des co-achats invisibles aux taxonomies.

Quel est le rôle du A/B test dans un pipeline de recommandation ?

Voir la réponse
Réponse

Le A/B test mesure l'impact causal du système de recommandation sur les métriques métier (conversion, panier moyen). Sans A/B test, on ne peut pas distinguer "les recommandations améliorent les ventes" de "les gens qui cliquent sur les recommandations achètent de toute façon plus".

Pourquoi la couche XGBoost de ranking final intègre-t-elle des features métier comme la marge produit ?

Voir la réponse
Réponse

Un système de recommandation pur maximise la pertinence pour l'utilisateur, pas la rentabilité pour l'entreprise. Le ranking XGBoost permet de pondérer la pertinence avec des objectifs commerciaux (marge, stock à écouler, partenariats). C'est la couche qui aligne ML et stratégie business.

43. Cas complet 2 — Prévision de ruptures de stock (Carrefour)

Le brief

Carrefour veut réduire les ruptures de stock en hypermarché. Coût actuel estimé : 4% du chiffre d'affaires perdu en ventes manquées. Reformulation : prédire la demande à 7 jours par référence produit et par magasin, pour déclencher des commandes anticipées.

Données disponibles : - Historique de ventes (2 ans, ~50M lignes) - Météo (historique + prévisions à 7 jours) - Calendrier (jours fériés, soldes, événements locaux) - Promotions planifiées (catalogue) - Données fournisseurs (délais de livraison, ruptures passées)

Architecture choisie

┌──────────────────────────────────────────────┐ │ PIPELINE PRÉVISION STOCK │ └─────────────────────┬────────────────────────┘ │ ┌─────────────────────────┼──────────────────────┐ ▼ ▼ ▼ ┌──────────────────┐ ┌───────────────────┐ ┌──────────────────┐ │ Prophet │ │ XGBoost │ │ Règles métier │ │ (tendance + │ │ (événements │ │ (seuils mini, │ │ saisonnalité │ │ exceptionnels : │ │ délais livraison│ │ par référence) │ │ météo, promos, │ │ par fournisseur)│ │ │ │ fériés) │ │ │ └────────┬─────────┘ └────────┬──────────┘ └──────────┬───────┘ │ │ │ └───────────────────────┼─────────────────────────┘ ▼ ┌────────────────────────┐ │ Prévision finale : │ │ Prophet + résidu XGB │ │ + contrainte règles │ └───────────┬────────────┘ ▼ ┌────────────────────────┐ │ MLflow (tracking) │ │ Airflow (orchestration) │ │ Ré-entraînement /semaine│ └────────────────────────┘

Pourquoi Prophet pour la base ? Prophet décompose proprement la tendance (croissance ou déclin d'une référence), la saisonnalité hebdomadaire (pic du samedi), et la saisonnalité annuelle (fêtes de fin d'année). Il gère nativement les jours fériés. Pour 80% des références sans événements particuliers, il est suffisant.

Pourquoi XGBoost pour les événements ? Prophet ne sait pas que la météo pluvieuse du week-end va faire exploser les ventes de raclette, ou que le catalogue de la semaine va tripler la demande de cet article spécifique. XGBoost apprend ces corrélations à partir des données historiques.

Pourquoi des règles pour les seuils ? Le stock minimum n'est pas une prédiction ML — c'est une contrainte contractuelle avec le fournisseur. On ne prédit pas le délai de livraison de 48h de Lactalis : on l'applique comme règle.

Code skeleton — Pipeline Prophet + XGBoost

from prophet import Prophet
import xgboost as xgb
import pandas as pd

def entrainer_pipeline(df_ref, ref_id):
    """Entraîne le pipeline pour une référence produit."""

    # Étape 1 : Prophet sur la tendance de base
    df_prophet = df_ref[['date', 'ventes']].rename(
        columns={'date': 'ds', 'ventes': 'y'}
    )
    modele_prophet = Prophet(
        yearly_seasonality=True,
        weekly_seasonality=True,
        holidays=jours_feries_france,  # DataFrame prédéfini
    )
    modele_prophet.fit(df_prophet)

    # Prévision base Prophet (7 jours)
    futur = modele_prophet.make_future_dataframe(periods=7)
    forecast = modele_prophet.predict(futur)

    # Étape 2 : XGBoost sur le résidu (événements exceptionnels)
    df_ref['residual'] = df_ref['ventes'] - df_ref['prophet_fitted']
    features_xgb = [
        'temperature_max', 'precipitation_mm',
        'est_promotion', 'est_ferie',
        'ventes_lag_7', 'ventes_lag_14',
        'semaine_annee', 'jour_semaine'
    ]

    modele_xgb = xgb.XGBRegressor(n_estimators=200, max_depth=5)
    modele_xgb.fit(df_ref[features_xgb], df_ref['residual'])

    return modele_prophet, modele_xgb

MLflow — outil de tracking d'expériences ML. Il enregistre automatiquement pour chaque entraînement : les hyperparamètres utilisés, les métriques obtenues (MAE, RMSE), les artefacts du modèle, et la version des données.

Airflow — outil d'orchestration de pipelines de données. Chaque dimanche à 2h, le DAG (Directed Acyclic Graph — graphe orienté sans cycle, représentant les étapes du pipeline) se déclenche : récupération des nouvelles données de vente, ré-entraînement, validation, déploiement si les métriques s'améliorent.

Métriques

  • Métrique technique : MAE (erreur absolue moyenne) par référence, sur les 7 jours suivants
  • Métrique métier : taux de rupture (objectif : passer de 4% à 1.5%) + coût de surstock évité
  • Monitoring : dérive de distribution des ventes (alerte si le MAE dépasse 2× le MAE historique)
Flashcards

Pourquoi combiner Prophet ET XGBoost plutôt que choisir l'un ou l'autre ?

Voir la réponse
Réponse

Prophet capture la structure temporelle régulière (saisonnalité, tendance). XGBoost capture les événements exceptionnels (météo, promotions) que Prophet ne peut pas utiliser car il n'intègre pas de covariables arbitraires facilement. L'architecture hybride exploite la complémentarité : Prophet pour la base, XGBoost pour les résidus.

Qu'est-ce que MLflow et à quel moment du cycle de vie ML intervient-il ?

Voir la réponse
Réponse

MLflow est un outil de suivi d'expériences. Il intervient pendant l'entraînement pour enregistrer : les paramètres utilisés, les métriques obtenues, et les artefacts du modèle. Sans MLflow, il est impossible de reproduire un résultat ou de comparer deux expériences plusieurs semaines plus tard.

Pourquoi les règles métier (seuils, délais fournisseurs) ne sont-elles pas remplacées par le modèle ML ?

Voir la réponse
Réponse

Les seuils de réapprovisionnement sont des contraintes contractuelles et logistiques, pas des patterns statistiques. Un modèle ML qui prédit 50 unités ne peut pas "apprendre" que le fournisseur livre en 72h minimum. Les règles encodent des connaissances déterministes ; le ML encode des corrélations probabilistes.

54. Cas complet 3 — Système hybride de traitement de réclamations (SNCF)

Le brief

La SNCF reçoit 12 000 réclamations par semaine via email, formulaire web et chatbot. Le traitement actuel prend 5 jours en moyenne. Objectif reformulé : réduire le délai de traitement à 48h en automatisant la classification, la priorisation et la génération de réponses standard pour les cas simples.

Données disponibles : - 3 ans de réclamations avec labels (catégorie, priorité, résolution) - Données structurées (retard en minutes, type de billet, gare, train) - Réponses rédigées par les agents (corpus de 50k exemples)

Contrainte critique : les décisions impactant des remboursements > 500€ ou des situations complexes doivent être validées par un humain.

Pourquoi CamemBERT et pas un LLM pour la classification ?

CamemBERT est un modèle BERT entraîné sur du texte français. Il produit une représentation vectorielle dense d'un texte, qu'on utilise comme entrée d'un classifieur.

Pourquoi pas directement GPT-4 pour classifier ? Trois raisons :

  1. Coût : classifier 12 000 emails par semaine avec GPT-4 coûte ~200€/semaine. CamemBERT fine-tuné coûte ~2€/semaine en inférence.
  2. Latence : CamemBERT répond en 50ms, GPT-4 en 2-3 secondes.
  3. Reproductibilité : un LLM peut changer de comportement entre deux versions d'API. CamemBERT est figé après le fine-tuning.
from transformers import CamembertTokenizer, CamembertForSequenceClassification
import torch

tokenizer = CamembertTokenizer.from_pretrained("camembert-base")
modele = CamembertForSequenceClassification.from_pretrained(
    "camembert-base",
    num_labels=5  # retard, annulation, remboursement, accident, autre
)

def classifier_reclamation(texte):
    inputs = tokenizer(texte, return_tensors="pt",
                       max_length=512, truncation=True, padding=True)
    with torch.no_grad():
        logits = modele(**inputs).logits
    proba = torch.softmax(logits, dim=-1)
    categorie = torch.argmax(proba).item()
    confiance = proba.max().item()
    return categorie, confiance

Le LLM uniquement pour la génération

Quand la confiance > 0.90 ET le montant < 500€, le LLM génère une réponse personnalisée :

import anthropic

client = anthropic.Anthropic()

def generer_reponse(reclamation, categorie, donnees_dossier):
    prompt = f"""Tu es un agent SNCF. Rédige une réponse professionnelle et empathique.

Catégorie de la réclamation : {categorie}
Détails du dossier : {donnees_dossier}
Réclamation du client : {reclamation}

Réponse (2-3 paragraphes, ton courtois, solution concrète proposée) :"""

    message = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=400,
        messages=[{"role": "user", "content": prompt}]
    )
    return message.content[0].text

Monitoring de la dérive

Dérive de concept (concept drift) : le sens des mots dans les réclamations change avec le temps. Exemple : "retard de connexion" signifiait train de correspondance en 2019 et WiFi embarqué en 2023. La distribution des catégories change aussi avec les incidents.

Indicateurs de dérive à surveiller : - Taux de confiance moyen de CamemBERT (baisse = dérive probable) - Distribution des catégories prédites (alerte si une catégorie explose ou disparaît) - Taux d'escalade humaine (augmentation = le modèle est moins sûr de lui)

Flashcards

Pourquoi CamemBERT est-il préféré à un LLM pour la classification de réclamations en production ?

Voir la réponse
Réponse

Trois raisons : coût (100× moins cher à l'inférence), latence (50ms vs 2-3s), reproductibilité (comportement stable après fine-tuning, un LLM peut changer entre versions d'API). Le LLM est réservé aux tâches génératives où sa flexibilité est un avantage.

Qu'est-ce que la dérive de concept (concept drift) et comment la détecter ?

Voir la réponse
Réponse

La dérive de concept survient quand la relation entre les features et la variable cible change dans le temps (ex : "retard de connexion" change de sens). On la détecte par : baisse du taux de confiance moyen du modèle, changement de la distribution des prédictions, augmentation du taux d'escalade vers les humains.

Comment déterminer le seuil au-delà duquel un cas doit être escaladé vers un humain ?

Voir la réponse
Réponse

Le seuil combine plusieurs critères : confiance du modèle (< 0.90 = incertitude), enjeu financier (> 500€ = risque d'erreur coûteuse), complexité (priorité haute = contexte émotionnel). Ces seuils ne sont pas techniques — ils sont négociés avec les équipes métier et le service juridique.

Pourquoi ne pas utiliser un seul LLM pour l'ensemble du pipeline (classification + priorisation + réponse) ?

Voir la réponse
Réponse

Un LLM unique serait 50-100× plus cher, 20× plus lent, et moins fiable sur la classification (un BERT fine-tuné surpasse GPT-4 sur des tâches de classification courte en domaine spécialisé). L'architecture hybride utilise chaque outil là où il excelle : BERT pour comprendre, XGBoost pour scorer, LLM pour générer.

Les pièges qui attendent en vrai projet

:::warning ⚡ Piège 1 — La qualité des données réelles

En cours, les datasets sont propres. En entreprise, 40 à 60% du temps projet est de la préparation de données. Les colonnes "date" ont 4 formats différents dans le même fichier. Les valeurs manquantes ne sont pas NaN mais "N/A", "non renseigné", "-", et 0 utilisé comme valeur par défaut. Le pipeline scikit-learn ne plante pas sur ces cas : il produit un modèle silencieusement faux. :::

:::warning ⚡ Piège 2 — Le changement de specs à mi-projet

Après 6 semaines de travail sur la prédiction d'attrition client, le métier revient avec : "En fait, on voudrait aussi avoir une explication pour chaque client prédit." Ce n'est pas un détail — c'est potentiellement un changement d'algorithme complet. L'explicabilité doit être dans les requirements dès le départ. :::

:::warning ⚡ Piège 3 — Le conflit avec le SI existant

Le modèle est prêt. Le déploiement prend 3 mois parce que la DSI impose une validation de sécurité, que le système de production tourne sur Python 3.6 (incompatible avec votre bibliothèque), et que personne n'a pensé à demander l'accès au serveur de production dès le début du projet. :::

:::warning ⚡ Piège 4 — La résistance au changement

Un modèle de recommandation qui surpasse les règles métier de 15% peut être rejeté par l'équipe commerciale qui ne comprend pas "pourquoi la machine conseille ça". La technique est rarement le problème final d'un projet ML. L'adoption l'est souvent. :::

La compétence la plus rare

:::rule ✅ Règle d'or — Savoir dire non à un LLM quand une regex suffit

La compétence la plus rare en 2025 n'est pas de savoir utiliser un LLM — tout le monde peut appeler une API. C'est de savoir quand ne pas le faire. Un pattern de validation d'email se résout en une ligne de regex. Une classification de 5 catégories sur des textes courts se résout avec CamemBERT fine-tuné. Un LLM sur ces cas coûte plus cher, répond moins vite, et est moins fiable. La sophistication non justifiée est une dette technique déguisée en innovation. :::

Recommandations concrètes pour la suite

Projets personnels (0-3 mois) - Kaggle : compétitions "Getting Started" (Titanic, Housing Prices) pour pratiquer le pipeline complet. Ensuite, rejoindre une compétition active dans ton domaine (NLP ou tabular). - Reproduire un des 3 cas du Jour 7 avec un dataset public similaire (MovieLens pour la recommandation, données météo pour la prévision).

Ressources - Hands-On Machine Learning (Géron) — le meilleur livre pour ancrer scikit-learn et les fondements - Fast.ai (cours en ligne, gratuit) — deep learning par la pratique, approche top-down - Papers With Code (paperswithcode.com) — l'état de l'art par tâche, avec le code

En alternance chez Capgemini - Proposer de faire le POC technique sur un projet client : construire le premier modèle baseline, mesurer l'AUC, présenter les limites. C'est le meilleur apprentissage. - Documenter les décisions d'architecture (pourquoi XGBoost et pas un LLM) dans un ADR (Architecture Decision Record). Ça force la réflexion et ça montre la maturité.

Quelle est la compétence la plus rare en ML selon le programme, et pourquoi ?

Voir la réponse
Réponse

Savoir dire non à un LLM quand une solution plus simple suffit. Parce que tout le monde sait appeler l'API OpenAI ou Anthropic. Ce qui différencie un bon data scientist est la capacité à choisir l'outil proportionnel au problème : regex, règles, XGBoost, ou LLM selon le contexte.

Pourquoi l'explicabilité doit-elle être dans les requirements dès le début d'un projet ?

Voir la réponse
Réponse

Parce qu'elle contraint le choix de l'algorithme. Un modèle de deep learning n'est pas nativement explicable : ajouter SHAP après coup est une rustine. Si l'explicabilité est requise (RGPD, décision sur une personne, conformité), il faut partir sur une régression logistique ou un arbre de décision, pas sur un réseau de neurones.

Quelle est la proportion habituelle du temps projet consacrée à la préparation des données en entreprise ?

Voir la réponse
Réponse

Entre 40% et 60% du temps projet. Les datasets réels ont des formats incohérents, des valeurs manquantes encodées de façons multiples, des colonnes dupliquées, et des erreurs de saisie. Contrairement aux datasets de cours, ils ne sont jamais "prêts à l'emploi".

Quelle est la première chose à faire si un projet ML échoue à être adopté par les équipes métier ?

Voir la réponse
Réponse

Investiguer les raisons de la résistance au changement : manque de confiance dans les prédictions, incompréhension du fonctionnement, peur de perdre le contrôle. La solution est souvent une formation courte + des outils de visualisation des décisions + un mode dégradé où l'humain garde la main.
