Exercice 1 — Architecturer un système hybride

Pour chacun des 3 cas ci-dessous, proposez une architecture hybride ML + LLM + règles. Pour chaque proposition, indiquez : - Quel composant fait quoi (ML classique, LLM, règles) - Pourquoi ce découpage (pas l'inverse) - Le point de défaillance le plus probable

Cas A — Modération de contenu chez une plateforme d'e-commerce

Contexte : Une marketplace reçoit 500 000 nouvelles annonces par jour. Certaines contiennent du contenu illicite (produits contrefaits, objets interdits, arnaque). Les règles actuelles basées sur des mots-clés ont un taux de faux positifs de 12% (annonces légitimes bloquées) et ratent 8% des fraudes sophistiquées qui contournent les mots-clés.

Contraintes : - 500 000 annonces/jour — le budget est limité - Temps de traitement cible : < 2 secondes par annonce - Les décisions de bannissement définitif doivent être auditables - L'annonce contient : titre (texte), description (texte libre, 0-2000 chars), prix, catégorie, photos (1-10)

À proposer : schéma du pipeline avec les composants, justification du découpage.

:::problem Question piège : pourquoi ne pas envoyer toutes les annonces directement à un LLM multimodal (texte + images) ? :::

Cas B — Système de prêt immobilier chez une banque régionale

Contexte : La banque reçoit 2 000 demandes de prêt immobilier par mois. Chaque dossier contient : données structurées (revenus, charges, historique bancaire), pièces justificatives scannées (bulletins de salaire, avis d'imposition, relevés), et un formulaire libre où le demandeur décrit son projet.

Contraintes : - Décision réglementaire : le motif de refus doit être expliqué au client (RGPD, DSP2) - Les conseillers passent actuellement 45 min par dossier sur la saisie des pièces - Taux d'erreur de saisie manuelle : 3,2% - La décision finale d'octroi reste humaine (réglementation bancaire)

À proposer : pipeline d'assistance à l'instruction du dossier (pas la décision finale).

Cas C — Chatbot de support technique pour un éditeur de logiciel SaaS

Contexte : L'éditeur gère 15 000 tickets de support par mois. 60% sont des questions documentées (les réponses existent dans la documentation, les changelogs, les tickets précédents résolus). 30% sont des bugs nouveaux qui nécessitent une investigation. 10% sont des demandes de features.

Contraintes : - La documentation est mise à jour toutes les 2 semaines (nouvelles versions) - Les tickets précédents résolus sont une mine d'or non exploitée (200 000 tickets sur 5 ans) - Les clients Enterprise ont un SLA de 2h pour la première réponse - Le code source du produit ne doit pas être exposé au LLM (propriété intellectuelle)

À proposer : architecture complète du système de support automatisé.

Cas 3.1 — Extraction de numéros de facture

Une équipe déploie un pipeline RAG + LLM pour extraire des informations d'un format de facture interne strictement standardisé. Chaque facture a exactement la même structure, les mêmes libellés, les mêmes positions. Le LLM coûte 0,02€ par facture × 400 000 factures/an = 8 000€/an.

Quelle est l'alternative et quel en est le coût estimé ?

Cas 3.2 — Classification de sentiment sur des avis produits

Un e-commerce reçoit 10 000 avis clients par jour. Chaque avis est évalué par GPT-4o pour en extraire : sentiment (positif/neutre/négatif), catégorie (livraison, produit, service client), et un résumé en 1 phrase.

Quelles parties peuvent être faites sans LLM ? Laquelle justifie un LLM ?

Cas 3.3 — Prédiction de churn

Une startup SaaS utilise Claude pour analyser le comportement des utilisateurs (logs d'événements JSON) et prédire s'ils vont churner dans les 30 jours. Le LLM reçoit le JSON des 30 derniers jours d'activité de chaque utilisateur.

Quels sont les 3 problèmes fondamentaux de cette approche ?

Cas 3.4 — Détection d'anomalies dans des séries temporelles

Pourquoi est-ce architecturalement impossible, indépendamment du coût ?

:::callout-insight Réponses Checkpoint

R1 : (1) LLM extrait les entités du texte (circonstances, dommages, tiers). (2) XGBoost score la probabilité de fraude sur features structurées + entités. (3) Règles métier décident (seuils). (4) Monitoring : faithfulness, hallucination rate, dérive du score fraude.

R2 : 500 documents → acceptable (calcul Python en < 100ms). 500 000 documents → inacceptable (charger 500k × 1536 floats en mémoire + calcul = plusieurs secondes par requête, insoutenable en prod). FAISS est conçu exactement pour ce problème.

R3 : (1) Enrichir le prompt système avec une instruction explicite "Réponds uniquement à partir des sources fournies, cite-les". (2) Augmenter k (récupérer plus de chunks) pour que le contexte pertinent soit plus souvent présent. Bonus : revoir le chunking (chunks trop grands = contexte dilué).

R4 : Horaire → Claude Haiku / GPT-4o-mini (question simple, réponse courte, coût minimal). Comparaison Enterprise/Pro → Claude Opus / GPT-4o (raisonnement multi-critères, qualité > coût). Prix produit doc → Claude Haiku + RAG (retrieval suffit, le LLM n'a besoin que de retranscrire).

R5 : Toujours utiliser des règles pour (1) les décisions à fort enjeu irréversibles (remboursement, bannissement, transaction financière), (2) les contraintes réglementaires qui doivent être auditables (motif de refus de prêt, RGPD). Exemples : seuil de montant pour validation humaine obligatoire ; catégories de produits interdits (liste explicite, pas un LLM). :::
